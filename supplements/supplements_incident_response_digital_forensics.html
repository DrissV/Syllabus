Session 1 : Threat Landscape & Bases fondamentales :

Trois périmètres à distinguer :

  1. Information Security : Garantir la Confidentialité, l'Intégrité et la Disponibilité (CID ou CIA en anglais) de l'information - quelle que soit sa forme.

  2. IT Security : Protéger les systèmes informatiques et les données contre les accès non autorisés (sous-ensemble de l'InfoSec).

  3. Cybersecurity : Défendre les systèmes connectés contre les attaques malveillantes - focus sur la menace cyber externe.

Vulnérabilité, Menace & Risque :

  • Vulnérabilité : Faiblesse dans un système, un processus, ou une organisation qui peut être exploitée.

    Exemples : patch de sécurité manquant, mot de passe faible, configuration incorrecte.

  • Menace (Threat) : Événement potentiel capable d'exploiter une vulnérabilité et de causer un préjudice.

    Exemples : ransomware, phishing, attaque DDoS, insider malveillant.

  • Risque : Probabilité qu'une menace exploite une vulnérabilité * Impact potentiel.

    RISQUE = VRAISEMBLANCE * IMPACT.

CVE & CVSS - Identifier et scorer les vulnérabilités :

CVE (Common Vulnerabilities and Exposures) :

Identifiant unique pour chaque vulnérabilité publique : CVE-ANNÉE-NUMÉRO.

Référentiel maintenu par MITRE, alimenté par les éditeurs et chercheurs.

CVSS (Common Vulnerability Scoring System) :

Score de 0 à 10 basé sur : vecteur d'attaque, complexité, impact CIA.

7-8.9 = Élevé. 9-10 = Critique. Exemples : Log4Shell = 10, Spectre = 5.6.

Ressources clés :

https://nvd.nist.gov

https://www.cvedetails.com

Session 2 : Hacking Methodology :

Les sept étapes d'une attaque :

  1. Information Gathering

  2. Scanning

  3. Enumeration

  4. Gaining Access (Intrusion)

  5. Privilege Escalation & Lateral Movement

  6. Covering Tracks

  7. Creating Backdoors

→ Chaque étape sera détaillée individuellement dans la suite du cours.

Note : ce schéma est une simplification pédagogique. En réalité, les attaques avancées (APT) ne sont pas linéaires - les étapes se chevauchent et se répètent.

Denial of Service (DoS/DDoS) et Ransomware sont traités séparément car ce sont des objectifs finaux, pas des étapes systématiques.

  1. Information Gathering (Reconnaissance) :

    Objectif de l'attaquant :

    Collecter un maximum d'informationtions sur la cible AVANT toute interaction directe.

    Passive : OSINT (Open Source Intelligence) :

    • LinkdIn (organigramme, technologies utilisées, emails).

    • WHOIS, DNS records, job postings, GitHub (code source exposé).

    • Shodan (équipements connectés), Google dorks, certificats TLS (crt.sh).

    Active : sondage direct :

    Ping sweep, traceroute, tests de connexion - plus détectable.

  2. Scanning :

    Objectif :

    Identifier les systèmes actifs, ports ouverts, services en écoute, OS.

    Outils courants :

    • Nmap : scanner réseau de référence (ports, services, versions, OS fingerprinting).

    • Masscan : scanner ultra-rapide pour grands espaces d'adressage.

    • Nesssus / OpenVAS / Qualys : scanners de vulnérabilités.

    Ce que ça révèle :

    • Ports ouverts → services en écoute → versions logicielles → CVEs exploitables.

    • Cartographie du réseau : combien d'hôtes, sous-réseaux, pare-feux ?

  3. Enumeration :

    Objectif :

    Extraire des informations détaillées des services identifiés : comptes, partages, rôles.

    Cibles typiques :

    • Active Directory (AD) : utilisateurs, groupes, politiques GPO, relations de confiance.

    • SMB : partages réseaux accessibles, fichiers exposés.

    • SNMP : configuration réseau, interfaces, routes.

    • LDAP, HTTP, FTP, SSH : banners, configurations par défaut.

    Résultat :

    L'attaquant connaît maintenant la topologie interne, les comptes actifs et les partages accessibles.

  4. Gaining Access (Intrusion) :

    Méthodes d'exploitation :

    • Exploitation de vulnérabilités : Metasploit, exploits publics (Exploit DB), code custom.

    • Phishing / Spear-phishing → malware (dropper, RAT, ransomware).

    • Credential stuffing / brute force (mots de passe faibles ou leakés).

    • Exploitation d'accès légitimes : VPN, RDP, Citrix mal configurés.

    • Supply-chain : compromission d'un fournisseur de confiance.

    Frameworks d'exploitation :

    • Metasploit (open source) - Cobalt Strike (commercial, très utilisé par APT).

    • Mimikatz - extraction de Credentials depuis la mémoire Windows (LSASS).

  5. Privilege Escalation & Lateral Movement :

    Privilege Escalation (escalade verticale) :

    Passer d'un compte utilisateur standard à Admin/root/SYSTEM.

    Via : exploits kernel, mauvaises configurations, tokens volés, services mal configurés.

    Objectif final souvent : Domain Admin sur Active Directory.

    Lateral Movement (mouvement latéral) :

    Se déplacer d'un système à l'autre dans le réseau interne.

    Techniques : Pass-the-Hash, Pass-the-Ticket, Remote Services (RDP, WMI, PsExec).

    Chaque système compromis devient un nouveau point de départ.

    Distinction clé :

    Escalade = monter en privilèges sur UN système. Latéral = se propager sur D'AUTRES systèmes.

  6. Covering Tracks (Dissimulation) :

    Objectif :

    Effacer les traces de l'intrusion pour rester non détecté et compliquer l'investigation.

    Techniques courantes :

    • Effacement ou modification des journaux (event logs Windows, syslog Linux).

    • Modification des horodatages de fichiers (timestomping).

    • Désactivation des antivirus / EDR.

    • Utilisation de comptes légitimes pour masquer l'activité malveillante.

    • Chiffrement des communications C2 (Command & Control) dans du trafic HTTPS légitime.

    Impact pour le forensics :

    Compliquer ou empêcher la reconstitution de la timeline de l'incident.

    → D'où l'importance des logs centralisés (SIEM) et des backups immuables.

  7. Creating Backdoors (Persistance) :

    Objectif :

    Maintenir un accès long terme au système compromis, même après redémarrage ou reset de mot de passe.

    Mécanismes courants :

    • Création de comptes admin cachés dans Active Directory.

    • Tâches planifiées / services Windows malveillants.

    • Modification des DLL système (DLL hijacking) ou des binaires OS.

    • Implants dans le firmware (UEFI rootkits - très difficiles à détecter).

    • Modification des règles de pare-feu ou ACL réseau.

    Lien avec les APT :

    Les APT maintiennent des accès pendant des mois ou des années - phase C2 (Command & Control).

    SolarWinds : backdoor SUNBURST dans une mise à jour logicielle légitime.

Denial of Service (DoS) & DDoS :

Objectif de l'attaque :

Rendre un service inaccessible aux utilisateurs légitimes - impact sur la Disponibilité (D du CID).

Types :

  • Volumétrique : saturer la bande passante (amplification DNS/NTP, exemple : 3,47 Tbps record 2022).

  • Protocole : exploiter les faiblesses TCP/IP (SYN flood, Ping Of Death).

  • Applicatif (Layer 7) : cibler les ressources serveur (HTTP flood, Slowloris).

DDoS = Distributed DoS :

Utilise un botnet (milliers de machines infectées) pour démultiplier l'attaque.

DDoS-as-a-Service disponible sur le dark web pour quelques dizaines d'euros/heure.

Défenses :

CDN avec filtrage DDoS (Cloudflare, Akamai), scrubbing centers, rate limiting.

Ransomware - Mécanisme et impact :

Comment ça fonctionne :

  1. Accès initial (phishing, vuln, credential volé).

  2. Escalade + lateral movement → contrôle de l'AD, accès aux backups.

  3. Exfiltration des données sensibles (double extorsion).

  4. Déploiement du chiffrement sur tout le réseau simultanément.

  5. Demande de rançon + menace de publication des données volées.

Ransomware-as-a-Service (RaaS) :

Groupes comme LockBit, REvil, Conti : fournissent le logiciel + support aux 'affiliés'.

Modèle commercial sophistiqué : partage des revenus, service client, négociation.

Impact réel :

Coût moyen d'un incident ransomware (PME) : > 500k€ - rançon + coûts de récupération.

Étape Kill Chain Correspondance 7 étapes Objectif attaquant

Ce schéma est votre référence pour toute la session 3. Chaque étape sera détaillée dans la suite du cours.

Parallèle avec le forensics : la collecte de preuves (Digital Forensics) se fait principalement pendant les phases Identification et Containment - avant toute action corrective qui pourrait altérer les preuves.

Détection - Qui voit quoi et comment ?

  • Employés & utilisateurs :

    > 40% des incidents sont d'abord remarqués par un employé non-sécurité. Les campagnes de sensibilisation ont un impact direct.

  • Alertes systèmes & applications :

    Logs serveurs, alertes applicatives, métriques d'infrastructure anormales.

  • Security Monitoring (SOC) :

    • SIEM (Splunk, QRadar, MS Sentinel, ELK) - corrélation de logs multi-sources.

    • EDR/XDR (Tanium, CrowdStrike, MS Defender) - détection comportementale endpoint.

    • NGFW, IDS/IPS, proxies, WAF - alertes réseau.

  • Tiers (CERT, partenaires, clients) :

    Cyber Threat Intelligence (CTI), OSINT, Threat Hunting, disclosure responsable.

Vérification & Qualification :

Question centrale :

Cet événement est-il un incident de sécurité ou une fausse alerte ?

Définition d'un incident de sécurité :

Toute tentative ou acte réussi d'accès non autorisé, utilisation, divulgation, modification ou destruction d'informations ou de systèmes.

→ Impact négatif sur la Confidentialité, l'Intégrité ou la Disponibilité (CID).

Processus de qualification :

  • Recueillir les premiers éléments : logs, alertes, témoignages.

  • Évaluer : est-ce une activité autorisée ? Un test de pénétration planifié ? Un bug ?

  • Si incident confirmé : déclencher la procédure formelle → notifier le Incident Manager.

  • Si non : documenter et archiver pour référence future.

Identification - Comprendre le périmètre :

Objectif :

Délimiter précisément ce qui est affecté, comment l'incident s'est produit et depuis quand.

Actions concrètes :

  • Identifier tous les systèmes, comptes et données impliqués.

  • Analyser les logs réseau, EDR, SIEM pour reconstruire la timeline.

  • Contacter les utilisateurs impactés pour collecter des informations.

  • Identifier le vecteur d'entrée initial (Patient Zéro).

  • Évaluer si l'attaquant est encore actif dans le réseau.

Collecte de preuves forensiques (voir session 5) :

  • Capture mémoire des systèmes critiques AVANT de les isoler ou éteindre.

  • Images disques des systèmes compromis pour analyse ultérieure.

Containment - Limiter la propagation :

Objectif :

Arrêter la propagation de l'incident sans détruire les preuves.

Containment court terme (immédiat) :

  • Isoler les systèmes compromis du réseau (VLAN isolation, déconnexion physique).

  • Bloquer les comptes compromis et révoquer les tokens d'accès.

  • Bloquer les IOC connus sur le pare-feu, proxy, EDR (IPs, domaines, hashs).

Containment long terme :

  • Changer les mots de passe de tous les comptes impactées (AD complet si nécessaire).

  • Vérifier l'intégrité des systèmes non compromis.

  • Patch d'urgence sur la vulnérabilité exploitée.

Attention aux preuves forensiques :

Éteindre un système détruit la mémoire RAM - capturer la mémoire AVANT isolation si possible.

Éradication - Éliminer la menace :

Objectif :

Supprimer complètement la présence de l'attaquant du réseau.

Actions techniques :

  • Supprimer les malwares, fichiers malveillants et outils déposés.

  • Désactiver / supprimer les comptes compromis et backdoors.

  • Réinitialiser tous les credentials potentiellement compromis.

Actions organisationnelles :

  • Informer les employés de la menace et des comportements à éviter.

  • Notifier les partenaires et clients impactés.

  • Notifier les autorités compétentes (RGPD, NIS2 - délais légaux !).

Attention : l'éradication n'est pas la recovery :

Ne pas remettre en production avant d'être certain que la menace est éliminée.

Recovery - Reprise d'activité :

Objectif :

Remettre les systèmes en production de manière contrôlée et sécurisée.

Prérequis indispensables :

  • Éradication confirmée : aucune trace de l'attaquant détectée.

  • Vulnérabilités exploitées patchées ou mitigées.

  • Monitoring renforcé activé pour détecter toute réactivation.

Processus de recovery :

  • Restaurer depuis des backups sains et vérifiés (hash de l'image).

  • Prioriser les systèmes critiques pour le business (processus métier clé).

  • Tester avant de remettre en production : accès, intégrité des données, performances.

Recovery ≠ retour à l'état précédent :

C'est l'opportunité de corriger les faiblesses identifiées pendant l'incident.

Lessons Learned & Rapport d'incident :

Lessons Learned (bilan post-incident) :

  • Réunion dans les deux semaines suivant la clôture de l'incident.

  • Pas un exercice de blame - focus sur amélioration des processus.

  • Questions clés : Qu'est-ce qui a fonctionné ? Qu'est-ce qui a échoué ? Comment améliorer ?

  • Résultat : plan d'actions correctives avec responsables et délais.

Structure du rapport d'incident :

  1. Identification : date, nature, systèmes affectés.

  2. Résumé exécutif : ce qui s'est passé, impact, actions menées.

  3. Notification : parties prenantes informées (internes + régulateurs).

  4. Actions : détail de chaque phase IR avec preuves.

  5. Évaluation & enseignements : plan d'amélioration.

Session 4 : IR Plan & Cyber Threat Intelligence :

Incident Response Plan - Pourquoi ?

Il n'existe que deux types d'entreprises : celles qui ont été hackées et celles qui le seront. - Robert S. Mueller, Directeur FBI.

Un IR Plan bien préparé permet de :

  • Réduire le temps de détection et de réponse (Mean Time To Detect / Respond).

  • Limiter l'impact business (temps d'arrêt, coût, atteinte à la réputation).

  • Satisfaire aux obligations légales et régulatoires (RGPD, NIS2).

  • Éviter les erreurs sous stress qui aggravent l'incident.

Les principes clés sont :

  • Implication du top management : la sécurité est un risque business, pas un problème IT.

  • Copie offline du plan : accessible même si le réseau est chiffré.

  • Tester le plan régulièrement (exercices de simulation, tabletop exercises).

  • Maintenir le plan à jour après chaque incident et chaque changement d'infrastructure.

Cyber Threat Intelligence (CTI) :

Définition :

Connaître ce que font vos adversaires et utiliser cette information pour améliorer la prise de décision.

Les quatre niveaux de CTI sont :

  • Stratégique : tendances, acteurs, géopolitique → pour la direction et CISO.

  • Opérationnelle : campagnes en cours, contexte d'une attaque → pour les équipes sécurité.

  • Tactique : TTPs (Tactiques, Techniques, Procédures) → pour les équipes SOC.

  • Technique : IOC (indicateurs de compromission) → pour les outils de détection.

Pourquoi c'est important ?

  • Anticiper les menaces avant qu'elles frappent votre organisation.

  • Prioriser les défenses en fonction des menaces réelles pour votre secteur.

  • Accélérer la réponse aux incidents grâce à des informations contextuelles.

Indicateurs de compromission (IOC) :

Un IOC est un artefact observable indiquant qu'une compromission a eu lieu ou est en cours.

Les types d'IOC sont :

  • Adresses IP : serveurs, adresses de scan, exfiltration.

  • Domaines (FQDN) : par exemple *.contosomalicious.cm, domaines DGA (Domain Generation Algorithm).

  • URLs : serveurs de téléchargement de payload, pages de phishing.

  • Hashs cryptographiques (MD5, SHA-1, SHA-256) : empreintes de fichiers malveillants.

L'usage opérationnel des IOC permet :

  • le blocage automatique sur NGFW, proxys, EDR et solutions de filtrage DNS;

  • la mise en place de règles de corrélation dans le SIEM afin de détecter les connexions vers des IOC connus.

Les limites des IOC sont :

  • Les attaquants changent d'IOC facilement (nouvelle IP, nouveau domaine).

  • Les TTPs (comportements) sont plus durables.

MITRE ATT&CK TTP Framework (opérationnel) :

Rappel : Tactiques → Techniques → Procédures :

  • Tactique = objectif adversarial (par exemple : Persistence - maintenir l'accès).

  • Technique = méthode utilisée (par exemple : T1543 - Create or Modify System Process).

  • Procédure = implémentation spécifique (par exemple : Lazarus Group utilise T1543.003 via un service Windows nommé "svchost32").

L'usage CTI avec ATT&CK permet de :

  • Mapper un incident détecté aux techniques ATT&CK → identifier le groupe probable.

  • Vérifier si d'autres techniques du même groupe sont présentes dans votre réseau.

  • ATT&CK Navigator : visualiser la couverture de détection de votre SOC.

Exemple Belacorp :

  • Spear Phishing (T1566.001) → Credential Dumping Mimikatz (T1003.001) → Lateral via PsExec (T1021.002) → Ransomware Impact (T1486).

Plateformes et partage d'information :

  • MISP (Malware Information Sharing Platform) est une plateforme open source de partage de CTI et d'IOC entre organisations. Elle est utilisée par CERT.be, les ISAC sectoriels ainsi que de nombreuses grandes entreprises.

  • OpenCTI est une plateforme open source de gestion et de visualisation de la CTI, incluant les TTPs ATT&CK. Elle propose des intégrations natives avec MISP, STIX/TAXII et VirusTotal.

  • EclecticIQ est une solution commerciale de CTI offrant une interface intuitive et des flux de renseignement intégrés.

Les formats standards d'échange sont :

  • STIX (Structured Threat Information eXpression) : format JSON pour représenter les objets CTI.

  • TAXII (Trusted Automated eXchange of Intelligence Information) : protocole de transport permettant l'échange automatisé de données STIX.

Supplément Gouvernance : ISO/IEC 27001 & ISO/IEC 27005 :

ISO 27001 : Qu'est-ce que c'est :

L'ISO 27001 est une norme internationale pour la mise en place, le maintien et l'amélioration continue d'un Système de Management de la Sécurité de l'Information (SMSI / ISMS).

Elle est publiée par l'ISO et l'IEC. La dernière version est ISO/IEC 27001:2022.

Pourquoi c'est important ?

  • Cadre reconnu mondialement pour démontrer la maturité sécurité d'une organisation.

  • Certification auditée par un organisme accrédité (par exemple : Bureau Veritas, SGS, BSI).

  • Reconnue comme référence dans le contexte NIS2 (§ Annexe II).

La famille ISO 27000 :

  • 27001 = exigences ISMS.

  • 27002 = guide des contrôles.

  • 27005 = gestion des risques.

  • 27035 = IR management.

ISO 27001 - Structure : Clauses 4 à 10 :

  • Clause 4 - Contexte de l'organisation : Parties prenantes, périmètre du ISMS, enjeux internes et externes.

  • Clause 5 - Leadership : Engagement de la direction, politique de sécurité, rôles et responsabilités.

  • Clause 6 - Planification : Évaluation des risques, plan de traitement, objectifs de sécurité.

  • Clause 7 - Support : Ressources, compétences, sensibilisation, communication, documentation.

  • Clause 8 - Opérations : Mise en oeuvre du plan de traitement des risques, gestion des incidents, continuité.

  • Clause 9 - Évaluation des performances : Surveillance, audits internes, revues de direction.

  • Clause 10 - Amélioration : Non-conformités, actions correctives, amélioration continue (PDCA).

ISO 27001:2022 - Annexe A : 93 contrôles en 4 thèmes :

  • Thème 1 - Organisationnels (37 contrôles) : Politiques, rôles, gestion des actifs, classification, fournisseurs, IR, continuité, conformité.

  • Thème 2 - Humains (8 contrôles) : Sélection, formation, sensibilisation, responsabilités, licenciement.

  • Thème 3 - Physiques (14 contrôles) : Périmètre de sécurité, accès physique, équipements, supports.

  • Thème 4 - Technologiques (34 contrôles) : Identités, accès, chiffrement, logs, conformité, vulnérabilités, développement sécurisé.

Les évolutions ISO 27001:2022 vs 2013 sont :

  • 11 nouveaux contrôles : threat intelligence (5.7), cloud security (5.23), data masking (8.11), secure coding.

  • La Statement of Applicability (SoA) documente l'applicabilité de chaque contrôle + justification.

Statement of Applicability (SoA) :

Définition :

Une SoA est un document listant TOUS les contrôles de l'Annexe A avec pour chacun : applicable (oui/non), justification, état d'implémentation, référence aux preuves.

Structure typique d'une ligne SoA :

N° contrôle | Intitulé | Applicable ? | Justification | Statut | Référence procédure

Exemple - Contrôle 5.7 Threat Intelligence :

  • Applicable : Oui.

  • Justification : exigence NIS2 Art. 21 + risque APT identifié dans l'évaluation.

  • Statut : Partiellement implémenté (abonnement CERT.be).

  • Référence : PRO-CTI-01.

Importance dans la certification :

  • Document audité en priorité lors d'un audit de verification ISO 27001.

  • Doit être cohérent avec l'évaluation des risques - chaque contrôle exclu doit être justifié.

  • Doit être maintenu à jour à chaque évolution du périmètre ou des risques.

Processus de certification ISO 27001 :

Étape 1 - Préparation (6-18 mois) :

  • Définir le périmètre, réaliser le gap analysis, mettre en place les contrôles manquants.

  • Produire les documents obligatoires : politique de sécurité, SoA, évaluation des risques, plan de traitement.

Étape 2 - Audit Stage 1 (document review) :

  • L'auditeur vérifie la documentation ISMS - typically 1-2 jours, sur site ou à distance.

  • Résultat : liste de points à clarifier avant le Stage 2.

Étape 3 - Audit Stage 2 (audit de certification) :

  • Vérification sur site de l'implémentation réelle des contrôles - entretiens, observations, preuves.

  • Durée : 2-10 jours selon la taille de l'organisation.

Étape 4 - Surveillance et renouvellement :

  • Audits de surveillance annuels - Renouvellement de la certification tous les 3 ans.

  • Organismes accrédités en Belgique : Bureau Veritas, SGS, Lloyd's, LRQA, BSI.

ISO 27001 et la gestion des incidents - Contrôles clés :

  • Contrôle 5.25 - Évaluation et décision des événements de sécurité → Processus de qualification (vérification) des incidents - Session 3.

  • Contrôle 5.26 - Réponse aux incidents de sécurité → Exécution de la procédure IR complète - Sessions 3 et 4.

  • Contrôle 5.27 - Apprentissage des incidents → Lessons Learned, amélioration continue - directement lié au PDCA.

  • Contrôle 5.28 - Collecte de preuves → Digital Forensics, Chain of Custody, intégrité des preuves - Sessions 5 et 6.

  • Contrôle 5.7 - Threat Intelligence → CTI, MISP, MITRE ATT&CK - Session 4.

ISO/IEC 27005 - Gestion des risques de sécurité :

Le rôle d'ISO 27005 est de fournir les lignes directrices pour la gestion des risques de sécurité de l'information, en soutien d'ISO 27001 (Clause 6).

La dernière version est ISO/IEC 27005:2022.

Les concepts fondamentaux d'ISO 27005 sont :

  • Actif (Asset) : tout ce qui a de la valeur pour l'organisation (données, systèmes, personnes, processus).

  • Vulnérabilité : faiblesse exploitable d'un actif.

  • Menace (Threat) : cause potentielle d'un incident non désiré.

  • Impact : conséquence d'un incident sur les objectifs de l'organisation.

Risque = f(Actif, Vulnérabilité, Menace, Vraisemblance, Impact)

Les liens avec d'autres frameworks sont :

  • Compatible avec ISO 31000 (gestion des risques générale), EBIOS Risk Manager (FR/BE), FAIR (quantification financière).

ISO 27005 - Processus d'évaluation des risques :

  1. Établissement du contexte : Périmètre, critères de risque, tolérance, parties prenantes.

  2. Identification des risques : Actifs - Menaces - Vulnérabilités - Conséquences.

  3. Analyse des risques : Vraisemblance * Impact → Niveau de risque.

  4. Traitement des risques : quatre options (Réduire, Accepter, Transférer, Éviter).

  5. Acceptation du risque résiduel : Validation par la direction - risques résiduels documentés.

  6. Communication & surveillance : Reporting régulier, revue annuelle minimum.

Identification et classification des actifs :

Les types d'actifs (exemples) sont :

  • Informationnels : base de données clients, données financières, propriété intellectuelle, contrats.

  • Logiciels : applications métier, OS, licences, code source.

  • Physiques : serveurs, postes de travail, équipements réseau, supports amovibles.

  • Humains : compétences clés, rôles critiques, connaissances tacites.

  • Services : cloud, télécommunications, alimentation électrique, climatisation.

La classification de la sensibilité est :

  • Public

  • Interne

  • Confidentiel

  • Strictement confidentiel (ou équivalent selon l'organisation)

Le registre des actifs (Asset Register) est un document central du ISMS qui liste tous les actifs avec propriétaire, valeur, classification, localisation.

Analyse menaces & vulnérabilités (27005) :

Catalogues de menaces :

  • ISO/IEC 27005:2022 - Annexe A & B : listes de menaces et vulnérabilités de référence.

  • ENISA Threat Landscape (ETL) - rapport annuel des menaces européennes.

  • MITRE ATT&CK - techniques d'attaque documentées (vu en session 2).

  • CERT.be - bulletins de menaces spécifiques au contexte belge.

Sources de vulnérabilités :

  • CVE/NVD - vulnérabilités techniques connues.

  • Résultats de scans (Nessus, Qualys, OpenVAS).

  • Résultats de tests d'intrusion (pentest).

  • Audits de configuration (CIS Benchmarks).

  • Retours d'incidents passés (lessons learned).

Options de traitement du risque - Les 4T :

Option Description Exemple Condition d'usage
Réduire (Treat / Mitigate) Mettre en place des contrôles pour diminuer vraisemblance ou impact Déployer MFA → réduit le risque de compromission credential Risque élevé, contrôle faisable et proportionné
Accepter (Tolerate / Accept) Décider consciemment de vivre le risque tel quel Risque de panne d'alimentation = faible, UPS coûteux Risque dans les limites de tolérance, coût de mitigation > impact
Transférer (Transfer) Déplacer le risque vers un tiers (assurance, sous-traitance) Assurance cyber pour couvrir le coût d'un ransomware Impact financier élevé mais vraisemblance faible
Éviter (Terminate / Avoid) Cesser l'activité qui génère le risque Ne pas traiter certaines données sensibles si inutile Activité dont le risque dépasse systématiquement la valeur

Registre des risques (Risk Register) :

Document vivant du ISMS :

Centralise tous les risques identifiés avec leur évaluation, traitement et suivi.

Structure type d'une entrée :

  • ID | Actif concerné | Menace | Vulnérabilité | Vraisemblance | Impact | Niveau de risque brut

  • Option de traitement | Contrôle(s) appliqué(s) | Risque résiduel | Propriétaire | Échéance | Statut

Exemple - Risque Belacorp (post-incident) :

  • ID : R-045

  • Actif : Active Directory

  • Menace : Credential Theft

  • Vuln : Absence MFA

  • Vraisemblance : Élevée

  • Impact : Critique → Risque brut : CRITIQUE

  • Traitement : Réduire

  • Contrôle : Déploiement MFA Azure AD

  • Risque résiduel : MODÉRÉ

  • Responsable : DSI

  • Délai : J+30

Utilisation pratique :

  • Revu lors des revues de direction trimestrielles/annuelles.

  • Mis à jour après chaque incident, pentest, audit ou changement majeur.

ISO 27001 ↔ NIS2 - Alignement et complémentarité :

Points de convergence :

  • ISO 27001 Art. 6 (évaluation des risques) ↔ NIS2 Art. 21.1 (gestion des risques).

  • ISO 27001 contrôles 5.24-5.28 (IR) ↔ NIS2 Art. 21.2(b) (traitement des incidents).

  • ISO 27001 contrôle 5.23 (sécurité cloud) ↔ NIS2 Art. 21.2(g) (sécurité chaîne d'approvisionnement).

ISO 27001 certification n'implique pas conformité NIS2 :

  • NIS2 ajoute des exigences spécifiques : délais de notification, sectoriel, responsabilité dirigeants.

  • Mais une organisation certifiée ISO 27001 a une large base de conformité NIS2.

Tableau de correspondance officiel :

  • ENISA a publié un mapping ISO 27001:2022 ↔ NIS2 disponible sur enisa.europa.eu.

  • En Belgique, le CB reconnaît ISO 27001 comme preuve partielle de conformité NIS2.

Session 5 : Digital Forensics : Principes & Collecte :

Digital Forensics - Définition et contexte :

Définition :

Investigation scientifique menée à la suite d'un incident ou d'une alerte de sécurité, visant à collecter, préserver et analyser des preuves numériques.

Deux contextes d'application :

  • Judiciaire : preuves destinées à un tribunal, standards de recevabilité stricts.

  • Interne / Corporate : analyse pour comprendre l'incident et améliorer les défenses - moins formel mais mêmes bonnes pratiques.

Trois types d'acquisition :

  • Live acquisition : capture de données volatiles sur un système en fonctionnement (mémoire RAM, connexions réseau).

  • Physical acquisition : image bit-à-bit du disque/SSD - système éteint, write-blocker obligatoire.

  • Memory acquisition : capture de la RAM volatile - essentielle pour les malwares fileless.

Principes fondamentaux : Intégrité & Documentation :

  1. Intégrité des preuves :

    • Ne jamais travailler sur l'original - toujours analyser une copie exacte (image forensique).

    • Utiliser des hash (MD5, SHA-256) pour vérifier l'intégrité à chaque étape.

  2. Minimiser l'impact sur les données :

    • Lors d'une acquisition live, l'acte de collecte modifie le système - documenter ces modifications.

    • Principe de Locard : tout contact laisse une trace - y compris l'investigation.

  3. Documentation exhaustive :

    • Documenter CHAQUE action : qui, quoi, quand, sur quel système, avec quel outil, quel résultat.

    • Un investigateur doit pouvoir expliquer et justifier chaque action en cours si nécessaire.

  4. Légalité de la collecte :

    • Ne collecter des preuves que sur des dispositifs où l'utilisateur a signé un règlement (RGPD + contrat de travail).

    • En Belgique : accord du propriétaire ou mandat judiciaire requis.

Chain of Custody - Traçabilité des preuves :

Définition :

Document traçant chaque personne ayant eu la garde d'une preuve, à chaque instant, depuis la collecte jusqu'à la restitution ou destruction.

Pourquoi c'est critique ?

  • Une preuve sans chain of custody peut être déclarée irrecevable en justice.

  • Elle garantit que la preuve n'a pas été modifiée, perdue ou altérée.

Informations requises dans le document :

  • Case ID, numéro et description de la preuve (marque, modèle, numéro de série, état physique).

  • À chaque transfert : coordonnées + date/heure + signature du cédant ET du recevant.

  • Hash de l'image forensique pour prouver l'intégrité à chaque étape.

Consentement du propriétaire :

L'utilisateur doit signer un document de remise volontaire avant la saisie de son appareil.

Naming Convention - Organiser les preuves :

Pourquoi standardiser les noms ?

Un incident implique souvent des dizaines de preuves. Sans convention, la confusion est garantie.

Structure recommandée :

  • Dossier : IncidentID_Prenom_Nom (par exemple : INC01254_Jean_Dupont).

  • Appareil saisi : Prenom_TypeAppareil (par exemple : Jean_PC001, Jean_MOBILE001).

  • Image d'acquisition : IncidentID_Prenom_AppareilID_SupportID.

  • Exemple complet : INC01254_Jean_Dupont_PC001_SSD001.

Règles pratiques :

  • Pas d'espaces ni de caractères spéciaux (compatibilité multiplateforme).

  • Inclure un numéro séquentiel si plusieurs images d'un même support (parties).

  • Documenter la convention choisie dans le rapport d'incident.

Types d'acquisition forensique :

Live Acquisition :

  • Système en fonctionnement.

  • Capture : mémoire RAM (volatile), connexions réseau actives, processus en cours.

  • Cas d'usage : malware fileless, attaquant actif, système critique qu'on ne peut éteindre.

  • Outils : WinPmem, Dumpit, FTK Imager (Windows)/linpmem (Linux)/osxpmem (Mac).

  • Risque : l'acte de collecte modifie légèrement l'état du système.

Physical Acquisition :

  • Système éteint (ou snapshot VM).

  • Capture : image bit-à-bit complète du disque/SSD.

  • Write-blocker hardware OBLIGATOIRE pour ne pas modifier les métadonnées.

  • Outils : FTK Imager, dd (Linux/Mac), Guymager.

  • Pour les VMs : snapshot + copie des fichiers .vmdk/.vhdx.

  • Formats d'images : .E01 (Expert Witness), .AD1, .dd (raw) - inclut les espaces libres.

Order of Volatility - Application pratique :

Décision : Live acquisition ou Physical acquisition ?

  • Système critique qui ne peut PAS être éteint → Live acquisition (mémoire + disque à chaud).

  • Attaquant potentiellement encore actif → Capture mémoire EN PREMIER pour capturer ses outils.

  • Malware fileless suspecté → Capture mémoire OBLIGATOIRE avant toute action.

  • Système éteint ou pouvant l'être → Physical acquisition (write-blocker, image bit-à-bit).

Commandes de collecte rapide (Windows - à lancer en admin) :

  • État réseau : netstat -anob > netstat.txt

  • Processus actifs : tasklist /v /fo csv > processes.txt

  • Services : sc query state=all > services.txt

  • Utilisateurs connectés : query user > sessions.txt

  • Partages ouverts : net session > netsessions.txt

Important: documenter l'heure exacte de chaque collecte (horodatage UTC).

Légalité et consentement - Cadre belge :

Bases légales pour la collecte de preuves :

  • Appareils de l'employeur : collecte autorisée si règlement de travail + politique IT signés par l'employé.

  • Appareils personnels de l'employé (BYOD) : accord écrit de l'employé ou mandat judiciaire.

  • Systèmes tiers (prestataires, partenaires) : accord contractuel ou mandat judiciaire.

Cadre légal belge :

  • Loi du 28 novembre 2000 relative à la cybercriminalité (intégrée dans le Code pénal belge).

  • RGPD / AVG : toute collecte de données personnelles doit être proportionnée et documentée.

  • Code d'instruction criminelle :seule la police judiciaire/magistrats peuvent saisir sous contrainte.

Recommandations pratiques :

  • Avoir une politique d'acceptable use (AUP) et de forensics intégrée dans le règlement de travail.

  • Documenter systématiquement le consentement AVANT la collecte.

  • Contacter le service juridique avant toute investigation d'insider threat.

Session 6 : Digital Forensics : Analyse & Timeline :

Artefacts Windows - Ce que le disque raconte :

$MFT (Master File Table) :

  • Méta-données de chaque fichier : date de création/accès/modification, taille, emplacement physique.

  • Survient à la suppression. Les données peuvent être récupérées même si le fichier est "effacé".

Event Logs Windows :

  • Security.evtx : authentification, création de comptes, modification de privilèges.

  • System.evtx : services démarrés/arrêtés, événements OS critiques.

Prefetch (C:\Windows\Prefetch) :

  • Traces de chaque application exécutée - timestamp + chemin d'exécution + fréquence.

  • Essentiel pour prouver qu'un malware ou outil a été exécuté, même s'il a été supprimé ensuite.

Registre Windows :

  • HKCU\Software\Microsoft\Windows\CurrentVersion\Run : programmes au démarrage (persistance).

  • UserAssist, RecentDocs, ShellBags : traces d'utilisation de l'interface.

Artefacts Linux & Mac : Points clés :

Les artefacts principaux de Linux sont :

  • /var/log : auth.log (authentification SSH, sudo), syslog, apache/nginx access.log.

  • ~/.bash_history : historique des commandes shell (attention : peut être effacé).

  • /proc : informations sur les processus actifs (live acquisition).

  • Cron jobs (/etc/crontab, /var/spool/cron) : tâches planifiées - vecteur de persistance courant.

  • Dernière connexion : /var/log/wtmp (tous logins), /var/log/btmp (tentatives échouées).

Les artefacts spécifiques de macOS sont :

  • Unified Log System : log show --last 24h - logs centralisés très détaillés.

  • LaunchDaemons/Agents : /Library/LaunchDaemons/, ~/Library/LaunchAgents/ -persistance.

  • Spotlight metadata (.DS_Store, plist) : activité fichiers, recherches, accès récents.

  • FileVault : chiffrement disk par défaut - nécessite les credentials ou la clé de récupération.

Reconstruction de Timeline - Méthode :

Objectif : Reconstituer chronologiquement les actions de l'attaquant de l'intrusion initiale jusqu'à la détection.

Les sources de timestamps sont :

  • Artefacts filesystem ($MFT, logs) : Created/Modified/Accessed/Entry-Modify (MACB).

  • Event logs : Security, System, Application, PowerShell.

  • Logs réseau : pare-feu, proxy, SIEM, DNS.

  • EDR : télémétrie comportementale temps réel.

  • Emails, VPN, applications métier : timestamps d'authentification et d'action.

La méthodologie de corrélation est :

  • Normaliser les fuseaux horaires (UTC recommandé) - un serveur NTP synchronisé est précieux.

  • Créer un tableau : Timestamp | Source | Événement | Système | Compte

  • Identifier les gaps : que s'est-il passé dans les plages sans données ?

  • Outils : Plaso/log2timeline, Timesketch (Google), Autopsy.

Threat Hunting - Chercher l'invisible :

Définition : Processus proactif de recherche de menaces cachées qui ont contourné les contrôles de détection automatisés.

Principe : ~80% des menaces sont détectées par les outils automatiques (EDR, SIEM). ~20% restent non détectées - le threat hunting vise ce résidu.

Les quatre étapes du Threat Hunting sont :

  1. Hypothèse : formuler une théorie sur une menace possible (par exemple : "Et si un attaquant APT28 était présent ?").

  2. Collecte : rassembler les données pertinentes (logs, artefacts, télémétrie EDR).

  3. Analyse : rechercher les indicateurs de l'hypothèse (TTPs ATT&CK du groupe).

  4. Conclusion : confirmer ou infirmer - documenter et améliorer les détections automatiques.

Différence avec l'IR : IR = réactif (répondre à une alerte). Threat Hunting = proactif (chercher sans alerte préalable).

Rapport forensique - Structure :

Deux audiences, deux documents :

  • Rapport exécutif (2-3 pages) : pour la direction, les juristes, les assureurs - sans jargon technique.

  • Rapport technique détaillé : pour les équipes sécurité et forensics - tous les détails.

Structure du rapport technique :

  1. Résumé : incident, timeline de haut niveau, conclusion principale.

  2. Méthodologie : outils utilisés, preuves trouvées, approche, chain of custody.

  3. Findings : artefacts analysés, preuves trouvées, timeline détaillée.

  4. Attribution (si possible) : liens avec des groupes ou campagnes connus.

  5. Recommandations : actions correctives techniques et organisationnelles.

  6. Annexes : hashes des preuves, logs bruts, captures d'écran.

Principes rédactionnels :

  • Factuel, objectif, sans spéculation non étayée.

  • Chaque affirmation est accompagnée de la preuve correspondante (référence à l'annexe).

NIS2 - Contexte et champ d'application :

Qu'est-ce que NIS2 ?

  • Directive européenne 2022/2555 - successor de NIS1 (2016). Transposée en droit belge fin 2024 (Loi CCB).

  • Élargissement majeur : ~160000 entités en Europe concernées (vs. ~15000 sous NIS1).

Deux catégories d'entités :

  • Entités Essentielles (EE) : énergie, transport, santé, eau, infrastructure numérique, spatial, finance.

  • Entités Importantes (EI) : services postaux, déchets, chimie, alimentation, fabrication, fournisseurs numériques.

Seuil d'application :

  • Entreprises moyennes (> 50 employés ou > 10M€ CA) dans les secteurs listés.

  • Certaines entités critiques quelle que soit leur taille (opérateurs infrastructure critique).

Belacorp et NIS2 :

250 employés, secteur logistique (transport) → potentiellement Entité Importante.

NIS2 - Obligations et notification d'incident :

Mesures de sécurité imposées (Article 21) :

  • Politique de sécurité des systèmes d'information et gestion des risques.

  • Réponse aux incidents + analyse forensique + continuité d'activité.

  • Sécurité de la chaîne d'approvisionnement (fournisseurs, sous-traitants).

  • Authentification forte (MFA), chiffrement contrôle des accès.

Obligation de notification d'incident (Article 23) :

  • Early Warning : dans les 24h - alerte initiale à l'autorité (CCB en Belgique).

  • Incident Notification : dans les 72h - rapport avec évaluation initiale de l'impact.

  • Final Report : dans le mois - rapport complet avec causes, impact, mesures prises.

Ce qui déclenche l'obligation de notification :

Impact significatif sur la continuité des services ou sur d'autres organisations.

RGPD Article 33 - Violation de données personnelles :

Qu'est-ce qu'une violation de données (data breach) ?

Tout incident entraînant la destruction, perte, altération, divulgation ou accès non autorisé à des données personnelles.

Obligation de notification à l'APD (Belgique) :

  • Dans les 72 heures suivant la prise de connaissance de l'incident.

  • Même en cas de doute - notifier sans attendre la certitude.

  • Via le portail : www.apd-gba.be.

Ce que doit contenir la notification :

  • Nature de la violation (type de données, nombre approximatif de personnes).

  • Coordonnées DPO (Délégué à la Protection des Données).

  • Conséquences probables et mesures prises ou envisagées.

Notification aux personnes concernées :

  • Obligatoire si risque élevé pour les droits et libertés des personnes.

  • Sans délai injustifié - en langage clair, accessible.

Recevabilité légale des preuves numériques :

Conditions générales de recevabilité :

  • Légalité : collecte autorisée (consentement, règlement de travail, mandat judiciaire).

  • Intégrité : preuves non altérées - démontré par les hashes et la chain of custody.

  • Authenticité : la preuve est ce qu'elle prétend être - documentation de la collecte.

  • Fiabilité : méthodes et outils scientifiquement reconnus et reproductibles.

Cadre pénal belge :

  • Code pénal belge, article 550bis : intrusions dans les systèmes informatiques.

  • Loi du 25 décembre 2016 sur la cybercriminalité : saisie de données informatiques.

  • Seuls les officiers de police judiciaire peuvent procéder à des saisies sous contrainte.

Expert judiciaire vs. expert interne :

  • Expert judiciaire : mandaté par le juge, rapport recevable en cour, accréditation requise.

  • Expert interne / DFIR : investigation pour améliorer les défenses, rapport interne.

Types de preuves devant un tribunal :

Classification générale des preuves :

Preuve directe :

  • Établit directement un fait sans inférence - par exemple : vidéo de surveillance montrant l'acte, aveux.

  • En cybersécurité : rarement disponible - les attaques laissent des traces indirectes.

Preuve circonstancielle (Indiciaire) :

  • Permet d'inférer un fait - par exemple : logs prouvant une connexion depuis un IP appartenant au suspect.

  • La majorité des preuves numériques sont circonstancielles - leur force vient de leur accumulation.

Preuve physique (Matérielle / Réelle) :

Objet tangible - en numérique : le disque dur, le serveur, la clé USB.

Preuve documentaire :

  • Documents présentés comme preuve - logs, emails, fichiers, captures d'écran.

  • Admissibilité conditionne la recevabilité : Authenticité + intégrité + légalité de collecte.

Témoignage :

Déclarations orales de témoins - inclut le témoignage d'expert (vu ci-après).

Preuves numériques - Conditions d'admissibilité :

Les cinq critères d'admissibilité (ACFLOW) sont :

  1. Authenticité : la preuve est ce qu'elle prétend être - démontrée par la chain of custody et les hashes.

  2. Complétude : les données collectées représentent fidèlement les faits - pas de sélection partielle.

  3. Fiabilité : méthodes et outils reconnus scientifiquement, reproductibles par un tiers.

  4. Légalité : collecte dans le cadre légal (consentement, mandat judiciaire).

  5. Objectivité : l'expert n'a pas d'intérêt personnel dans le résultat - impartialité requise.

Les problèmes courants d'admissibilité sont :

  • Preuve collectée sans consentement ni mandat → irrecevable.

  • Hash d'intégrité absent ou non vérifié → authenticité contestable.

  • Outil forensique non reconnu ou résultats non reproductibles → fiabilité contestée.

  • Chain of custody incomplète → continuité de la preuve contestée.

Témoignage d'expert vs. témoin ordinaire :

Témoin ordinaire (Témoin des faits) :

  • Témoigne uniquement de ce qu'il a directement observé - "j'ai vu l'alerte à 3h17".

  • Ne peut pas exprimer d'opinions ou d'interprétations.

Expert judiciaire (Gerechtsdeskundige) :

  • Nommé par le juge - inscrit au registre des experts judiciaires (SPF Justice belge).

  • Peut exprimer des opinions professionnelles et des conclusions analytiques.

  • Obligations : impartialité, exhaustivité, transparence sur les méthodes et limites.

  • Son rapport fait foi jusqu'à preuve du contraire.

Expert de partie (Private Expert) :

  • Désigné par une partie (plaignant ou défense) - rapport utilisable mais contestable.

  • En Belgique, le rapport d'un expert de partie a moins de poids qu'un rapport d'expert judiciaire.

Standards de rapport d'expert :

  • Méthodologie transparente.

  • Conclusions limitées aux faits prouvés.

  • Limites clairement énoncées.

Objectifs :

Permettre à l'étudiant de fournir une réponse approprié aux incidents dans l'entreprise, ainnsi qu'entreprendre les démarches forensiques adéquates.

Définitions :

  • La sécurité de l'information est la pratique qui consiste à garantir la confidentialité, l'intégrité et la disponibilité des informations.

  • La sécurité informatique (IT Security) est la pratique visant à protéger les données et les systèmes d'information contre les accès non autorisés. Elle couvre des domaines comme la sécurité des systèmes, la sécurité des réseaux, etc.

  • La cybersécurité est la pratique consistant à défendre les ordinateurs, les serveurs, les réseaux et les données contre les attaques malveillantes.

  • Vulnérabilité : qualité ou état d'un système exposé à la possibilité d'être attaqué ou endommagé, que ce soit physiquement ou logiquement.

  • Menace (Threat) : possibilité d'une tentative malveillante visant à endommager ou perturber un réseau ou un système informatique.

  • Risque (Risk) : situation impliquant une exposition à un danger ou à une menace.

  • Vulnérabilité zero-day : une vulnérabilité dans un système ou un appareil qui est inconnue des personnes chargées de la corriger (et donc non encore corrigée).

  • La surface d'attaque correspond à l'ensemble des vulnérabilités exploitables pour mener une attaque de sécurité, qu'elles soient physiques ou numériques.

  • Un acteur de la menace (Threat actor/agent) est une entité (personne, groupe ou organisation) responsable, en partie ou totalement, d'un incident ayant un impact ou pouvant impacter la sécurité d'une organisation.

  • Le paysage des menaces (Threat landscape) désigne l'ensemble des caractéristiques, des méthodes d'attaque et des objectifs des différents types de menaces auxquels une organisation est exposée.

  • La criminalistique (Forensics) regroupe l'ensemble des techniques et méthodes scientifiques utilisées pour détecter, analyser et prouver des faits liés à une activité criminelle.

  • L'informatique légale (Digital Forensics) consiste à collecter, préserver, analyser et présenter des preuves numériques (ordinateurs, réseaux, téléphones, etc.) dans le cadre d'une enquête.

Exemples de vulnérabilités : Mitre Common Vulnerabilities and Exposures (CVE).

La Security Risk Matrix (Matrice des risques de sécurité) est un outil qui permet d'évaluer et de prioriser les risques en fonction de leur probabilité et de leur impact.

Risque = Probabilité x Impact : Le risque est le produit de la probabilité (likelihood) qu'un événement se produise et de son impact (les conséquences) s'il se réalise.

Examen blanc :

Session 1 : Threat Landscape & bases fondamentales :

Question 1

Quel est l'objectif principal de la sécurité de l'information (Information Security) ?

  1. Protéger uniquement les serveurs connectés à Internet. ❌

  2. Garantir la confidentialité, l'intégrité et la disponibilité des informations. ✅

  3. Empêcher toutes les cyberattaques externes. ❌

  4. Protéger les systèmes informatiques contre les accès non autorisés. ⚠

La sécurité de l'information repose sur le modèle CIA (Confidentialité, Intégrité, Disponibilité). Elle vise à protéger l'information dans son ensemble, pas uniquement contre les attaques externes.

Limiter les accès non autorisés est un objectif de la confidentialité, mais cela ne couvre ni l'intégrité ni la disponibilité.

Question 2

Qu'est-ce qu'une vulnérabilité ?

  1. Une faiblesse pouvant être exploitée par une menace. ✅

  2. Une attaque déjà réalisée avec succès. ❌

  3. Une mesure de sécurité préventive. ❌

  4. Une menace susceptible d'affecter un système. ⚠

Une vulnérabilité est une faiblesse technique, humaine ou organisationnelle pouvant être exploitée pour compromettre un système.

Une menace peut exploiter une vulnérabilité, mais elle ne constitue pas elle-même une vulnérabilité.

Question 3

Comment calcule-t-on le risque ?

  1. Menace * Vulnérabilité. ❌

  2. Impact * Nombre d'actifs concernés. ⚠

  3. Vraisemblance * Impact. ✅

  4. CVSS * CIA. ❌

Le risque est classiquement défini comme la combinaison de la probabilité d'occurence et de l'impact.

Le nombre d'actifs influence l'impact global, mais ne remplace pas la notion de probabilité.

Question 4

À quoi sert le CVSS ?

  1. Identifier les groupes APT. ❌

  2. Évaluer la gravité d'une vulnérabilité. ✅

  3. Déterminer l'origine géographique d'une attaque. ❌

  4. Prioriser les actions de sécurité d'une organisation. ⚠

CVSS est un standard permettant de donner un score de gravité à une vulnérabilité.

La priorisation est une conséquence de l'évaluation CVSS, mais pas son objectif principal.

Question 5

Pourquoi Stuxnet est-il considéré comme historique ?

Le Digital Forensics consiste à collecter, préserver et analyser des preuves numériques dans un cadre d'enquête.

La reconstruction de la chronologie est une étape importante de l'analyse, mais elle ne représente pas l'ensemble du processus forensique.

  1. Pour chiffrer la preuve. ❌

  4. Pour identifier rapidement un fichier connu. ⚠

Le hash permet de vérifier qu'une preuve n'a pas été modifiée entre deux étapes d'analyse.

Les hash peuvent être utilisés pour identifier des fichiers connus, mais leur objectif principal reste la vérification d'intégrité.

Question 23

Que garantit la Chain of Custody ?

  1. La confidentialité des données collectées. ❌

  2. La disponibilité des preuves. ❌

  3. La traçabilité complète des manipulations effectuées sur une preuve. ✅

  4. La validité juridique des preuves. ⚠

La chain of custody documente toutes les étapes de manipulation d'une preuve pour garantir sa traçabilité.

Une bonne traçabilité contribue à la validité juridique, mais ne la garantit pas à elle seule.

  4. Les connexions réseau enregistrées dans un fichier log. ⚠

La RAM est une mémoire volatile qui perd son contenu lorsque le système s'éteint.

Les connexions réseau peuvent être temporaires, mais elles restent stockées sur disque et ne sont pas strictement volatiles.

  4. Pour garantir l'intégrité des preuves collectées. ⚠

Un write blocker empêche toute écriture sur le support afin de préserver son état original.

L'intégrité est un résultat direct du blocage en écriture, mais ce n'est pas la fonction technique elle-même.

Le Threat Hunting est une démarche proactive qui cherche à découvrir des activités malveillantes ayant échappé aux mécanismes de détection classiques.

L'activité proposée par la réponse approximative existe, mais elle correspond davantage à de la réponse à incident ou à une investigation post-alerte. Le caractère proactif du Threat Hunting disparaît ici.

Question 30

Quel principe est essentiel lors de la rédaction d'un rapport forensique technique ?

  1. Présenter les hypothèses les plus probables même lorsqu'aucune preuve ne les confirme. ⚠

Un rapport forensique doit être factuel, objectif et traçable. Chaque affirmation doit pouvoir être reliée à une preuve ou à un artefact identifié lors de l'investigation.

Dans une enquête forensique, les hypothèses sont utiles pendant l'analyse, mais elles ne doivent pas être présentées comme des conclusions sans preuves suffisantes. Une hypothèse plausible reste une hypothèse tant qu'elle n'est pas étayée par des éléments objectifs.

Le cours précise qu'il est nommé par le juge et inscrit au registre des experts judiciaires.

Question 33

Que constitue principalement un log montrant une connexion depuis une adresse IP attribuée à un suspect ?

Question 35

Une entreprise collecte secrètement les emails personnels d'un employé sans consentement, découvre une preuve de fraude, puis présente cette preuve au tribunal. Quel critère ACFLOW est le plus directement menacé ?

Les emails peuvent être parfaitement authentiques. Le problème n'est pas leur origine mais la manière dont ils ont été obtenus.

Questions d'entraînement (sessions 1 à 5) :

Question 41

Une organisation a identifié qu'un de ses serveurs web tourne sur une version PHP non supportée. Selon le modèle Vulnérabilité-Menace-Risque, qu'est-ce que la "version PHP non supportée" représente ?

  2. Un risque car cela crée une exposition pour l'organisation. ⚠

  3. Une vulnérabilité c'est une faiblesse exploitable dans un système. ✅

  4. Un impact car cela affecte la disponibilité du service. ❌

Une version PHP non supportée constitue une faiblesse technique exploitable. L'absence de correctifs de sécurité permet à un attaquant d'utiliser des vulnérabilités connues.

Le risque est la combinaison d'une menace exploitant une vulnérabilité avec un impact potentiel. La version PHP non supportée n'est pas encore le risque lui-même; elle est un élément qui contribue à créer ce risque.

Question 42

Un attaquant dispose d'un accès avec un compte utilisateur standard sur un poste Windows. Son objectif est de devenir "Domain Admin". Quelle technique utilise-t-il en priorité ?

  1. Lateral Movement : se déplacer vers un autre poste déjà Domain Admin. ⚠

  2. Privilege Escalation : exploiter une vulnérabilité locale pour monter en privilèges sur ce système. ✅

  3. Covering Tracks : effacer les logs pour ne pas être détecté avant d'agir. ❌

  4. Information Gathering : scanner le réseau pour trouver d'autres cibles. ❌

Avant d'obtenir des privilèges élevés, l'attaquant doit augmenter ses droits. L'escalade de privilèges est précisément la technique permettant de passer d'un compte limité à un compte plus puissant.

Le mouvement latéral permet de se déplacer vers d'autres machines. Cela peut effectivement mener à un Domain Admin si une machine compromise contient des identifiants privilégiés. Toutefois, la question demande la technique utilisée en priorité pour augmenter ses privilèges; c'est donc l'escalade de privilèges.

Question 43

WannaCry s'est propagé massivement en 2017 sans aucune interaction des victimes. Quelle capacité technique le rendait "auto-propageable" ?

  1. Il utilisait des emails de phishing à grande échelle avec des pièces jointes PDF. ❌

  2. Il exploitait EternalBlue (CVE-2017-0143) pour se propager via SMBv1 sans intervention humaine. ✅

  4. Il utilisait des credentials par défaut pour accéder aux systèmes non protégés. ⚠

WannaCry utilisait l'exploit EternalBlue contre SMBv1. Une fois une machine compromise, il recherchait automatiquement d'autres systèmes vulnérables.

L'utilisation d'identifiants par défaut est une technique qui peut favoriser la propagation d'un malware. Cependant, ce n'est pas le mécanisme utilisé par WannaCry.

Question 44

  1. Identifier les responsables de l'incident et documenter les sanctions disciplinaires. ❌

  2. Améliorer les processus, outils et compétences pour réduire la probabilité et l'impact d'incidents futurs. ✅

  4. Valider que l'éradication est complète avant de remettre les systèmes en production. ❌

La phase Lessons Learned vise l'amélioration continue; comprendre ce qui a fonctionné ou non afin de mieux réagir lors d'incidents futurs.

Les conclusions des Lessons Learned peuvent alimenter un rapport réglementaire, mais ce n'est pas l'objectif principal de cette réunion.

Question 45

Le principal intérêt d'un MSSP (Managed Security Service Provider ou prestataire de services de sécurité managés en français) est d'apporter des capacités de surveillance et de détection avancées sans devoir recruter une équipe SOC complète.

Un MSSP participe à la détection et parfois à la réponse aux incidents. Cependant, l'entreprise reste responsable de disposer de son propre plan de réponse à incident.

Question 46

Le nom indique qu'il s'agit d'un disque suspect et contient une date, mais il manque les identifiants précis nécessaires à la traçabilité.

Question 48

Un threat hunter émet l'hypothèse : "APT29 pourrait être présent dans notre réseau". Quelle est sa première action concrète selon la méthode en quatre étapes ?

  1. Alerter immédiatement la direction et déclencher la procédure IR complète. ❌

  2. Consulter MITRE ATT&CK pour identifier les TTP spécifiques d'APT29, puis rechercher ces signatures dans les données disponibles. ✅

  3. Scanner tous les systèmes avec un antivirus mis à jour pour confirmer ou infirmer l'hypothèse. ❌

  4. Contacter CERT.be pour obtenir des IOC actuels liés à APT29. ⚠

Le threat hunting repose d'abord sur une hypothèse puis sur la recherche de comportements (TTP) associés à l'acteur ciblé.

Les IOC du CERT.be peuvent être utiles, mais ils ne remplacent pas l'analyse comportementale fondée sur les TTP de l'acteur recherché.

Question 49

Une entreprise belge subit une intrusion. Des données personnelles de clients (noms, emails, numéros de téléphone) ont été exfiltrées. Quand l'obligation de notification à l'APD commence-t-elle ?

  1. Dès que l'entreprise a la certitude absolue que des données ont été exfiltrées. ⚠

  2. Uniquement si plus de 1000 personnes sont affectées. ❌

  3. Dès la prise de connaissance de la violation, le délai de 72h court immédiatement. ✅

  4. Après la clôture de l'incident, pour inclure toutes les informations dans la notification. ❌

Selon le RGPD, l'autorité compétente doit être notifiée dans les 72 heures après la prise de connaissance de la violation lorsqu'elle est susceptible d'engendrer un risque pour les personnes concernées.

Une certitude absolue n'est pas requise. Attendre la confirmation complète pourrait faire dépasser le délai de 72 heures.

Question 50

Un employé utilise son smartphone personnel (BYOD) pour accéder aux emails professionnels. Une investigation révèle des preuves potentielles d'insider threat sur cet appareil. Quelle est la procédure légalement correcte en Belgique ?

  1. Saisir immédiatement le smartphone avec l'accord du responsable IT. ❌

  2. Demander un accès distant via MDM (Mobile Device Management) pour extraire les données professionnelles. ⚠

  3. Obtenir le consentement écrit de l'employé ou un mandat judiciaire avant toute collecte. Un appareil personnel ne peut pas être saisi sans l'une ou l'autre condition. ✅

  4. Contacter directement l'opérateur télécom pour obtenir les logs de l'appareil. ❌

Un smartphone personnel reste la propriété de l'employé. Son examen nécessite une base légale valable, comme un consentement explicite ou une décision judiciaire.

Dans certaines organisations, le MDM permet l'accès à certaines données professionnelles présentes sur l'appareil. Toutefois, cela ne donne pas automatiquement le droit d'effectuer une collecte forensique complète sur un appareil personnel.

Questions supplémentaires :

Question 51

Quelle est la différence entre un hacker et un hacktiviste ?

  1. Un hacker travaille uniquement pour des entreprises, tandis qu'un hacktiviste travaille uniquement pour des gouvernements. ❌

  2. Un hacker crée des virus informatiques alors qu'un hacktiviste répare les ordinateurs infectés. ❌

  3. Un hacktiviste est généralement un hacker qui utilise ses compétences informatiques pour défendre une cause politique, sociale ou idéologique. ✅

  4. Les hacktivistes sont des hackers plus expérimentés qui ont obtenu une certification officielle. ⚠

Un hackviste utilise effectivement des techniques de piratage informatique dans le but de soutenir ou promouvoir une cause politique, sociale ou idéologique.

Les hacktivistes peuvent être expérimentés, mais il n'existe pas de certification officielle qui transforme un hacker en hacktiviste; c'est surtout la motivation de leurs actions qui les distingue.

Question 52

Quelle est la différence entre une attaque DoS et une attaque DDoS ?

  1. Une attaque DoS utilise une seule source pour saturer un service, tandis qu'une attaque DDoS utilise plusieurs machines coordonnées. ✅

  2. Une attaque DoS vise les ordinateurs et une attaque DDoS vise uniquement les sites web. ❌

  3. Une attaque DDoS est toujours légale, contrairement à une attaque DoS. ❌

  4. Une attaque DDoS est simplement une version plus rapide d'une attaque DoS. ⚠

Une DoS (Denial of Service) provient généralement d'une seule source, alors qu'une DDoS (Distributed Denial of Service) est menée depuis de nombreuses machines réparties sur différents réseaux.

Une DDoS peut être plus puissante qu'une DoS, mais la différence essentielle n'est pas la vitesse; c'est le fait qu'elle est distribuée sur plusieurs machines.

Question 53

Quelle est la différence entre le phishing et le spear phishing ?

  1. Le phishing se fait uniquement par e-mail, tandis que le spear phishing se fait uniquement par SMS. ❌

  2. Le spear phishing est simplement une version plus sophistiquée du phishing utilisant toujours l'intelligence artificielle. ⚠

  3. Le phishing cible un grand nombre de personnes avec le même message, tandis que le spear phishing cible une personne ou un groupe précis avec un message personnalisé. ✅

  4. Le phishing est légal alors que le spear phishing est illégal. ❌

Le phishing est une attaque de masse envoyée à de nombreuses victimes potentielles, alors que le spear phishing est ciblé et personnalisé pour une personne, une entreprise ou un groupe particulier.

Le spear phishing est souvent plus sophistiqué et personnalisé, mais il n'utilise pas nécessairement l'intelligence artificielle.

Question 54

Quelle est la principale différence entre ISO/IEC 27001 et ISO/IEC 27005 ?

  1. ISO 27005 est une version plus récente et plus complète d'ISO 27001 qui la remplace progressivement. ⚠

  2. ISO 27001 est obligatoire en Europe alors qu'ISO 27005 est facultative. ❌

  3. ISO 27001 définit les exigences pour mettre en place un SMSI, tandis qu'ISO 27005 fournit des lignes directrices pour la gestion des risques de sécurité de l'information. ✅

  4. ISO 27001 concerne uniquement les incidents de sécurité et ISO 27005 uniquement la sécurité physique. ❌

ISO 27001 décrit les exigences à respecter un Système de Management de la Sécurité de l'Information (SMSI), tandis qu'ISO 27005 explique comment réaliser l'évaluation et le traitement des risques exigés par la clause 6 d'ISO 27001.

ISO 27005 est bien complémentaire à ISO 27001, mais elle ne la remplace pas; elle fournit uniquement des recommandations pour la gestion des risques.

Question 55

Quelle est la différence entre un IOC et une TTP dans le contexte de la Cyber Threat Intelligence (CTI) ?

  1. Un IOC est utilisé uniquement par les dirigeants, tandis qu'une TTP est utilisée uniquement par les administrateurs système. ❌

  2. Les IOC servent à détecter les menaces et les TTP servent à chiffrer les données. ❌

  3. Un IOC est un indicateur observable de compromission, tandis qu'une TTP décrit le comportement ou la méthode utilisée par un attaquant. ✅

  4. Une TTP est essentiellement une liste d'adresses IP malveillantes plus détaillée qu'un IOC. ⚠

Un IOC (Indicator of Compromise) est un élément observable (IP, domaine, URL, hash, etc.) indiquant une compromission, tandis qu'une TTP (Tactic, Technique and Procedure) décrit la manière dont un attaquant agit pour atteindre son objectif.

Une TTP peut être associée à des IOC, mais elle représente surtout un comportement une technique d'attaque, pas une simple liste d'indicateurs techniques.

Question 56

Quelle est la différence entre STIX et TAXII dans le partage de Cyber Threat Intelligence (CTI) ?

  1. STIX est une plateforme de partage d'IOC et TAXII est une base de données de vulnérabilités. ❌

  2. TAXII est une version plus récente de STIX qui remplace progressivement ce format. ⚠

  3. STIX est un format permettant de représenter les informations de renseignement sur les menaces, tandis que TAXII est un protocole permettant d'échanger ces informations automatiquement. ✅

  4. STIX est utilisé uniquement pour les malwares et TAXII uniquement pour les attaques de phishing. ❌

STIX (Structured Threat Information eXpression) définit la structure des données CTI (IOC, TTP, acteurs de menace, campagnes, etc.), tandis que TAXII (Trusted Automated eXchange of Intelligence Information) est le protocole utilisé pour transporter et partager ces données entre plateformes.

STIX et TAXII sont complémentaires et conçus pour fonctionner ensemble; TAXII ne remplace pas STIX.

Question 57

Parmi les propositions suivantes, laquelle présente correctement les sept étapes classiques d'une cyberattaque ?

  1. Information Gathering → Encryption → Enumeration → Backup → Gaining Access → Monitoring → Creating Backdoors. ❌

  2. Information Gathering → Scanning → Gaining Access → Privilege Escalation → Covering Tracks → Creating Backdoors → Incident Response. ⚠

  3. Information Gathering → Scanning → Enumeration → Gaining Access → Privilege Escalation & Lateral Movement → Covering Tracks → Creating Backdoors. ✅

  4. Scanning → Information Gathering → Gaining Access → Antivirus Bypass → Enumeration → Covering Tracks → Reporting. ❌

Les sept étapes classiques d'une cyberattaque sont : collecte d'informations, analyse de la cible, énumération, intrusion, élévation de privilèges et mouvements latéraux, effacement des traces et maintien de l'accès.

Lors de la réponse approximative, plusieurs étapes sont présentes, mais l'Enumeration est absente. Incident Response est une activité défensive, pas une étape de l'attaque.

Question 58

Quelle est la différence entre CVE et CVSS dans la gestion des vulnérabilités ?

  1. CVE est un outil de test de sécurité et CVSS est un antivirus utilisé pour corriger les failles. ❌

  2. CVSS est un identifiant de vulnérabilité plus récent qui remplace progressivement les CVE. ⚠

  3. CVE est un identifiant unique pour une vulnérabilité, tandis que CVSS est un système de score qui évalue sa gravité. ✅

  4. CVE et CVSS sont deux systèmes de notation différents permettant tous deux de classer les attaques par type de malware. ❌

Le CVE (Common Vulnerabilities and Exposures) sert à identifier de manière unique une vulnérabilité (par exemple : CVE-2021-44228), tandis que le CVSS (Common Vulnerability Scoring System) attribue un score de gravité (0 à 10) basé sur l'impact et les caractéristiques techniques.

CVSS ne remplace pas CVE; ils sont complémentaires et utilisés ensemble dans l'écosystème de sécurité.

Question 59

Que signifie le PDCA ?

  1. Ping-DNS-Connect-Attack. ❌

  2. Processus-Deploy-Crash-Analyze ⚠

  3. Plan-Do-Check-Act pour amélioration continue du SMSI. ✅

  4. Protect-Detect-Control-Audit. ❌

Le PDCA (Plan-Do-Check-Act) est le cycle d'amélioration continue utilisé dans les systèmes de management comme l'ISO 27001 pour structurer, mettre en oeuvre, vérifier et améliorer en permanence le SMSI.

Même si les mots "Process" et "Deploy" peuvent vaguement évoquer des activités informatiques, la séquence "Processus-Deploy-Crash-Analyze" ne correspond à aucun cadre reconnu et mélange des notions opérationnelles sans logique standard de gestion ou d'amélioration continue comme le PDCA.

Question 60

Qu'est-ce qu'une image forensic ?

  1. Un log système. ❌

  2. Une copie bit-à-bit fidèle d'un support de stockage. ✅

  3. Un antivirus. ❌

  4. Une sauvegarde cloud. ⚠

Une image forensic est une copie bit-à-bit fidèle d'un support de stockage (disque dur, SSD, clé USB, etc.). Elle reproduit exactement les données, y compris les fichiers supprimés, les espaces libres et les artefacts système, afin de garantir l'intégrité des preuves lors d'une analyse forensique.

Une sauvegarde cloud peut contenir des données similaires à celles d'un système, mais elle n'est pas nécessairement une copie bit-à-bit complète ni destinée à préserver les preuves numériques dans un état forensiquement exploitable. Elle est pensée pour la restauration, pas pour l'investigation légale.

Questions traitées ensemble en classe :

Question 61

Stuxnet (2010) est souvent cité comme un tournant dans l'histoire de la cybersécurité. Pourquoi ?

  1. C'est le premier malware à avoir utilisé le chiffrement RSA pour ses communications C2. ❌

  2. C'est le premier ransomware d'État à avoir demandé une rançon à un gouvernement souverain. ❌

  3. C'est le premier cyberweapon documenté ayant causé des dommages physiques réels dans le monde physique. ✅

  4. C'est le premier malware à s'être propagé via une supply chain logicielle (mise à jour compromise). ⚠

Stuxnet a saboté des centrifugeuses nucléaires iraniennes en modifiant le comportement d'automates industriels. C'est le cas emblématique d'une cyberattaque provoquant des dégâts matériels réels.

Stuxnet utilisait plusieurs vecteurs de propagation (USB, réseau Windows, etc.) mais n'est pas considéré comme le premier exemple majeur d'attaque par supply chain logicielle. Cette caractéristique n'est pas ce qui a marqué l'histoire de Stuxnet.

Question 62

Lors d'une phase d'énumération sur un réseau Windows, quel outil permet de cartographier les chemins d'escalade de privilèges dans Active Directory ?

  1. Wireshark capture et analyse le trafic réseau pour identifier les authentifications Kerberos. ⚠

  2. Nmap scanne les ports et services de tous les contrôleurs de domaine. ❌

  3. BloodHound analyse les relations AD et visualise les chemins vers Domain Admin. ✅

  4. Metasploit exploite les vulnérabilités identifiées dans le contrôleur de domaine. ❌

BloodHound est précisément conçu pour cartographier les relations de confiance, délégations et privilèges dans Active Directory.

Wireshark peut effectivement observer du trafic Kerberos, mais il ne cartographie pas les chemins d'escalade de privilèges dans Active Directory.

Question 63

Un RSSI (Responsable de la Sécurité des Systèmes d'Information, équivalent du CISO) rédige un rapport d'incident pour la direction générale. Quel principe doit guider la rédaction du résumé exécutif ?

  1. Inclure tous les détails techniques pour montrer la profondeur de l'investigation. ⚠

  2. Éviter de mentionner les failles organisationnelles pour protéger l'image de l'équipe sécurité. ❌

  3. Être factuel, sans jargon technique, en articulant clairement l'impact business et les actions prises. ✅

  4. Copier le résumé exécutif du rapport RGPD pour l'APD afin d'assurer la cohérence. ❌

Le comité de direction veut comprendre les conséquences métier, les risques et les décisions à prendre, pas les détails techniques.

Les détails techniques ont leur place dans le rapport complet mais pas dans un résumé destiné à la direction.

Question 64

Un RSSI reçoit un rapport CTI indiquant que des groupes APT ciblent le secteur logistique avec des campagnes de spear-phishing via des portails de suivi de colis falsifiés. À quel niveau de CTI appartient cette information ?

  1. Technique, car elle peut être traduite en IOC (URL, domaines de faux portails). ❌

  2. Stratégique, car elle informe sur les tendances de menaces pour la direction. ❌

  3. Opérationnel, car elle décrit une campagne en cours avec son vecteur et son ciblage sectoriel. ✅

  4. Tactique, car elle décrit les TTPs (spear-phishing via portail) utilisés par les attaquants. ⚠

L'information décrit une campagne active, son ciblage (logistique) et son mode opératoire général. C'est typiquement du renseignement opérationnel.

Le spear-phishing constitue effectivement une TTP, mais l'information va plus loin en décrivant une campagne ciblée en cours; cela relève donc davantage du niveau opérationnel.

Question 65

Une entreprise certifiée ISO 27001:2022 est-elle automatiquement conforme à NIS2 ?

  1. Oui, ISO 27001 couvre tous les articles de NIS2 applicables aux entreprises. ❌

  2. Oui, la certification ISO 27001 est reconnue par le CCB comme équivalente à la conformité NIS2 complète. ❌

  3. Non, ISO 27001 fournit une large base de conformité, mais NIS2 ajoute des exigences spécifiques non couvertes (délais de notification, responsabilité dirigeants, exigences sectorielles). ✅

  4. Partiellement, ISO 27001 couvre NIS2 pour les aspects techniques mais pas pour les aspects de gouvernance. ⚠

ISO 27001 aide fortement à satisfaire NIS2 mais ne couvre pas automatiquement les obligations réglementaires de notification, de gouvernance ou de responsabilité des dirigeants.

La réponse approximative est l'inverse du problème : ISO 27001 contient justement beaucoup de gouvernance. L'affirmation contient une part de vérité (la couverture n'est pas totale) mais l'explication est erronée.

Question 66

Quel outil est le plus approprié pour capturer la mémoire RAM d'un serveur Windows 2022 en production ?

  1. dd, l'outil Unix universel pour les copies bit-à-bit. ❌

  2. FTK Imager Lite peut capturer la RAM et le disque depuis une interface graphique. ⚠

  3. WinPmem ou Dumpit, outils légers dédiés à la capture mémoire Windows sans installation requise. ✅

  4. Volatility Framework analyse la mémoire et peut aussi la capturer directement. ❌

Les outils WinPmem ou Dumpit sont spécifiquement conçus pour l'acquisition mémoire Windows en contexte forensic.

FTK Imager peut effectivement acquérir la RAM, mais il n'est généralement pas considéré comme l'outil de référence le plus léger et le plus spécialisé pour cette tâche.

Question 67

Une organisation découvre qu'un prestataire externe a accédé illégalement à ses systèmes depuis son ordinateur personnel. L'organisation veut analyser cet ordinateur. Quelle est la procédure correcte ?

  1. L'organisation peut saisir l'ordinateur car le prestataire a signé un contrat de service. ❌

  2. L'organisation contacte la FCCU (Federal Computer Crime Unit) pour procéder à une saisie judiciaire. ✅

  3. L'organisation demande à son RSSI de se connecter à distance à l'ordinateur du prestataire pour collecter les preuves. ❌

  4. L'organisation notifie d'abord l'APD avant toute collecte car des données personnelles sont impliquées. ⚠

L'ordinateur appartient au prestataire. Une entreprise privée ne peut pas saisir ou forensiquement examiner un équipement personnel sans base légale appropriée.

Une notification RGPD peut devenir nécessaire selon les circonstances, mais elle ne donne aucun droit d'accès à l'ordinateur du prestataire et ne résout pas la question de la collecte de preuves.

Question 68

Un attaquant pratique le timestomping pour modifier les dates du $MFT de ses fichiers malveillants. Quel artefact Windows est le plus résistant à cette technique ?

  1. Les dates MACB du $MFT sont stockées en doublon et l'une d'elles ne peut pas être modifiée. ❌

  2. Le journal des événements Windows (Security.evtx) : les dates d'entrée ne peuvent pas être modifiées. ⚠

  3. Les entrées Prefetch enregistrent les timestamps d'exécution indépendamment du $MFT. ✅

  4. Le fichier hiberfil.sys conserve les timestamps originaux lors de l'hibernation. ❌

Le timestomping vise généralement les timestamps du système de fichiers. Les artefacts Prefetch conservent des informations d'exécution souvent indépendantes des dates manipulées dans le MFT.

Les journaux Windows peuvent fournir des indices précieux mais ils peuvent être effacés, désactivés ou ne pas contenir les événements recherchés. Ils ne constituent pas une protection spécifique contre le timestomping.

Question 69

Une entreprise de livraison de colis belge (500 employés, 80M€ de CA) souhaite savoir si elle est soumise à NIS2. Quelle analyse est correcte ?

  1. Non, NIS2 ne s'applique qu'aux entreprises dépassant 250M€ de CA. ❌

  2. Oui, comme Entité Essentielle - le transport est un secteur critique de niveau 1. ⚠

  3. Oui, probablement comme Entité Importante - le transport est un secteur couvert et l'entreprise dépasse les seuils d'une entreprise moyenne. ✅

  4. Non, seules les entreprises publiques ou para-publiques sont soumises à NIS2 en Belgique. ❌

Le transport fait partie des secteurs couverts par NIS2 et une entreprise de 500 employés dépasse largement les seuils d'une entreprise moyenne.

L'entreprise est probablement concernée par NIS2, mais rien dans l'énoncé ne permet d'affirmer qu'elle relève automatiquement de la catégorie des entités essentielles.

Question 70

Une entreprise belge subit un ransomware avec exfiltration de données clients. Les systèmes sont tombés pendant 36h. À quelles autorités doit-elle notifier et dans quels délais ?

  1. Uniquement à l'APD dans les 72h - NIS2 ne s'applique qu'aux infrastructures critiques de niveau national. ⚠

  2. Au CCB(Early Warning 24h + notification 72h) et à l'APD (notification 72h) - les deux obligations courent en parallèle. ✅

  3. D'abord à l'APD dans les 72h, puis au CCB dans les 30 jours - les délais sont séquentiels. ❌

  4. Au CCB dans les 24h uniquement - la notification RGPD n'est requise que si plus de 5000 personnes sont affectées. ❌

Si l'organisation est soumise à NIS2 et qu'il y a violation de données personnelles, les obligations NIS2 et RGPD s'appliquent simultanément.

La réponse approximative serait correcte pour une organisation non concernée par NIS2, mais l'énoncé vise justement le cumul des obligations NIS2 et RGPD.
