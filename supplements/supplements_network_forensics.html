Section 1 : Introduction, concepts clés et enjeux :

Objectifs :

  • Comprendre ce qu'est la forensique réseau.

  • Identifier les différences entre les types de forensique.

  • Cerner les enjeux juridiques et opérationnels.

  • Appréhender le cycle d'investigation numérique réseau.

Définition - Network Forensics :

  • Branche de l'investigation numérique

  • Analyse du trafic réseau pour identifier et répondre à des incidents de sécurité.

Contexte d'utilisation :

  • Réponse à incident (IR).

  • Enquêtes judiciaires / internes.

  • Analyse post-mortem d'attaques.

  • Recherche de preuves de compromission (IOC).

Network vs Disk Forensics (Data Analysis) :

  • Données volatiles vs persistantes.

  • Sources : PCAP, NetFlow vs Disques, RAM.

  • Objectif : reconstruction vs extraction.

Exemples concrets :

  • Analyse d'un ransomware (trafic vers C2).

  • Détection d'un DNS tunneling.

  • Compromission via phishing.

Aspects légaux & éthiques :

  • RGPD / ePrivacy / Autorité de protection des données.

  • Traçabilité & consentement.

  • Chaîne de possession (chain of custody).

  • Confidentialité des communications.

Cycle d'investigation :

  1. Identification (alertes, anomalies).

  2. Collecte (pcap, logs, netflow).

  3. Analyse (outils, reconstruction).

  4. Corrélation (autres sources).

  5. Rapport & préservation.

Outils d'analyse réseau :

  • Wireshark, tcpdump, TZeek, Suricata, TShark.

  • Arkime, NetworkMiner, ELK, Velociraptor.

Quiz / Discussion / Réflexion :

  1. Quelles sont les limites de l'analyse réseau ?

    Les principales limites de l'analyse réseau sont :

    • Chiffrement (TLS 1.3, QUIC) : contenu souvent invisible.

    • Visibilité partielle : SPAN/TAP ne voient pas toujours tout.

    • Volumes énormes : difficile de tout capturer et stocker.

    • Cloud et Kubernetes : environnements dynamiques et éphémères.

    • Perte de contexte : le réseau voit les flux, mais pas toujours l'utilisateur ou le processus.

    • Faux positifs / faux négatifs : détection imparfaite.

    • Trafic interne (East-West) : parfois difficile à observer.

    • Coût et complexité : stockage des PCAP, sondes, performance.

    • Évasion des attaquants : tunnels, chiffrement, fragmentation, protocoles modernes.

    Aujourd'hui, l'analyse réseau seule ne suffit plus : elle doit être combinée avec :

    • EDR,

    • logs,

    • identité,

    • SIEM,

    • télémétrie des endpoints et du cloud.

  2. Peut-on faire du network forensics sans avoir les flux ?

    Oui, mais avec des limites importantes.

    Sans PCAP ou flux complets, on peut encore faire du network forensics à partir de :

    • logs firewall,

    • NetFlow/IPFIX,

    • DNS,

    • proxy,

    • VPN,

    • EDR,

    • journaux cloud,

    • événements SIEM.

    Cela permet souvent de :

    • reconstruire une chronologie,

    • identifier des communications suspectes,

    • détecter une exfiltration ou un C2,

    • attribuer des connexions.

    Mais sans les flux complets :

    • impossible de voir le contenu exact,

    • difficile d'analyser les payloads,

    • certaines preuves disparaissent,

    • la reconstruction est moins précise.

    Donc :

    • oui pour une investigation partielle et comportementale,

    • non pour une analyse réseau exhaustive.

  3. Quels sont les enjeux juridiques dans le cloud ?

    • Protection des données : respect du RGPD et des données personnelles.

    • Localisation des données : où les données sont stockées et transférées.

    • Accès par des autorités étrangères : par exemple, Cloud Act américain.

    • Responsabilités partagées : client vs fournisseur cloud.

    • Conservation des logs et preuves : exigences forensic et conformité.

    • Confidentialité et chiffrement : gestion des clés et accès aux données.

    • Multi-tenant : séparation entre les clients dans le cloud.

    • Conformité réglementaire : ISO 27001, NIS2, PCI-DSS, HIPAA, etc.

    • Chaîne de preuve : difficulté à garantir l'intégrité des preuves numériques.

    En pratique, le cloud impose de combiner :

    • sécurité,

    • gouvernance,

    • conformité,

    • contrats clairs avec le fournisseur.

Section 2 : Architecture réseau & points de collecte :

Objectifs :

  • Revoir l'architecture réseau (OSI & TCP/IP).

  • Identifier les différents points de collecte de données.

  • Comparer les méthodes de capture (SPAN vs TAP vs logs).

  • Comprendre les contraintes de déploiement et de performance.

Rappels modèles réseau :

  • Modèle OSI (7 couches) vs pile TCP/IP (4 couches).

  • Couches clés pour l'analyse : Réseau, Transport, Application.

  • Importance du contexte protocolaire pour la forensique.

Protocoles critiques à analyser :

  • ARP / NDP (résolution d'adresses).

  • DNS (résolution de noms, détection du tunneling).

  • HTTP/HTTPS (contenu & métadonnées).

  • SMTP/IMAP (phishing, exfiltration).

  • FTP/SFTP, SSH, RDP (accès distant).

Sources de données - Capture directe :

  • Port mirroring / SPAN sur switch.

  • TAP (Test Access Point) matériel.

  • Avantages : trafic complet, horodatage précis.

  • Inconvénients : coût, risques de goulot d'étranglement.

Sources de données - Logs & flux :

  • Logs pare-feu / UTM.

  • NetFlow / sFlow / IPFIX.

  • IDS/IPS (Suricata, Snort) alert logs.

  • Proxies TTP/NGFW.

  • Endpoint / EDR network telemetry.

Capture centralisée vs distribuée :

  • Centralisée : collecte au coeur du réseau ou datacenter.

  • Distribuée : sondes multiples (filiales, cloud, OT).

  • Avantages / inconvénients : visibilité vs complexité.

  • Considérations : latence, bande passante, chiffrement.

Placement stratégique des sondes :

  • Périmètre (edge) pour le trafic entrant/sortant.

  • DMZ pour services exposés.

  • Segment interne critique (serveurs, OT, IoT).

  • Liaisons inter-sites (VPN, MPLS).

  • Environnements cloud (VPC mirroring, packet capture).

Performance & stockage :

  • Débit et packet loss (capture à 1/10/40 Gbps).

  • Options de filtrage en temps réel (BPF, DPI).

  • Compression, déduplication.

  • Rétention des PCAP vs logs (coût stockage, législation).

  • Horodatage précis (PTP, NTP, GPS).

Quiz / Discussion / Réflexion :

  1. Comment choisir entre SPAN et TAP pour une investigation ?

    Le choix entre SPAN et TAP dépend surtout de ce que vous cherchez à observer, du niveau d'intrusion acceptable et du contexte réseau.

    La différence rapide est :

    • SPAN (Switched Port Analyzer, port mirroring) → Le switch copie le trafic vers un port d'analyse.

    • TAP (Test Access Point) → Un équipement matériel placé physiquement sur le lien qui reproduit le trafic.

    Quand choisir un SPAN ? Si :

    • Vous faites du troubleshooting ponctuel

    • Vous voulez une solution rapide et gratuite

    • Vous avez accès aux switches

    • Vous analysez un trafic modéré

    • La perte de quelques paquets n'est pas critique

    Les avantages de l'utilisation du SPAN sont :

    • Pas de matériel supplémentaire

    • Configuration rapide

    C'est très pratique pour :

    • Wireshark

    • IDS temporaire

    • debug applicatif

    • analyse réseau occasionnelle

    Le SPAN peut :

    • Perdre des paquets sous forte charge

    • Modifier légèrement les timings

    • Ne pas reproduire certaines frames/erreurs bas niveau

    • Être impacté par le CPU/ASIC du switch

    Quand choisir un TAP ? Si :

    • Vous faites de la capture critique

    • Vous avez besoin d'une visibilité 100% fidèle

    • Vous surveillez du trafic à haut débit

    • Vous faites :

      • forensic

      • détection d'intrusion

      • SOC/NDR

      • capture légale

      • analyse de performance précise

    Les avantages de l'utilisation d'un TAP sont :

    • Capture quasi parfaite

    • Très faible latence

    • Aucun impact logique sur le switch

    • Voit les erreurs physiques que SPAN peut manquer.

    C'est plus fiable pour :

    • Zeek

    • Suricata

    • full packet capture

    • appliances de sécurité

    Les inconvénients de l'utilisation d'un TAP sont :

    • Coût matériel

    • Installation physique

    • Peut nécessiter une brève coupure réseau

    • Plus complexe opérationnellement

    La règle pratique est :

    • Choisir SPAN si Je veux regarder ce qui se passe rapidement.

      Par exemple : debug VLAN, trafic suspect ponctuel, capture avec Wireshark et validation applicative.

    • Choisir TAP si Je dois être sûr de ne rien manquer.

      Par exemple : SOC, forensic, IDS/IPS passif, datacenter, backbone et environnements critiques.

    Critère SPAN TAP
    Coût Faible Plus élevé
    Déploiement Très simple Physique
    Fidélité Moyenne à bonne Excellente
    Perte de paquets Possible Très rare
    Visibilité erreurs L1/L2 Limitée Complète
    Impact switch Oui Non
    Usage permanent Moyen Excellent
    Forensic/SOC iLimté Recommandé

  2. Quels défis pose le chiffrement TLS 1.3 aux collectes ?

    TLS 1.3 complique fortement les collectes réseau car il chiffre beaucoup plus tôt et plus complètement mes communications.

    Les points clés sont :

    • Le contenu applicatif devient pratiquement invisible en capture passive.

    • La clé privée du serveur ne permet plus de déchiffrer les anciennes sessions (PFS/(EC)DHE).

    • Les métadonnées visibles diminuent (SNI, certificats, handshake).

    • Avec QUIC/HTTP3, l'analyse devient encore plus difficile.

    • Un TAP voit toujours tous les paquets, mais ils restent chiffrés.

    • Un SPAN subit les mêmes limites, avec en plus des risques de pertes.

    La conséquence est que les SOC et enquêteurs s'appuient davantage sur :

    • logs,

    • EDR,

    • NetFlow,

    • DNS,

    • analyse comportementale,

    • interception TLS active (proxy SSL).

  3. Comment capturer le trafic dans un environnement cloud ?

    Dans le cloud, la capture réseau se fait surtout via des mécanismes virtuels plutôt qu'avec des TAP physiques.

    Les principales approches sont :

    • Traffic Mirroring (AWS, Azure, GCP) : copie du trafic vers une sonde IDS/NDR.

    • Flow Logs : métadonnées réseau (IP, ports, volumes), sans payload.

    • Capture dans la VM : tcpdump/Wireshark pour du debug ciblé.

    • Agents eBPF/EDR : visibilité avancée avec contexte processus/workload.

    • Kubernetes : observabilité via Cilium, Hubble, service mesh, eBPF.

    Les défis majeurs sont :

    • trafic chiffré (TLS 1.3),

    • workloads éphémères,

    • volumes très élevés,

    • trafic East-West difficile à voir.

    En pratique, les environnements cloud modernes reposent davantage sur :

    • logs,

    • télémétrie,

    • métadonnées réseau,

    • eBPF,

    • observabilité applicative,

    • plutôt que sur la capture PCAP complète permanente.

Section 3 : Capture et collecte de données :

Objectifs :

  • Comprendre les méthodes de capture de trafic réseau.

  • Utiliser les outils de capture en ligne de commande et GUI.

  • Gérer le filtrage, le stockage et la rotation des captures.

  • Garantir l'intégrité et la traçabilité des preuves réseau.

Méthodes de capture réseau :

  • Capture en ligne : Wireshark, tcpdump, TShark.

  • Capture sur interface SPAN/TAP.

  • Utilisation de sondes : Zeek, Suricata, Arkime.

  • Capture dans le cloud : AWS VPC Mirroring, Azure Network Watcher.

Outils de capture - CLI :

  • tcpdump : outil léger, puissant, TShark - version en ligne de commande de Wireshark.

  • dumpcap - capture rapide avec Wireshark.

  • Filtres BPF : host, port, net, proto.

Outils de capture GUI :

  • Wireshark - visualisation complète - Affichage des couches OSI.

  • Filtres d'affichage vs filtres de capture - Coloration syntaxique pour anomalies.

Collecte passive vs active :

  • Passive : sans modification du trafic, préférée pour la forensique.

  • Active : injection ou manipulation (utile en tests mais intrusif).

  • La forensique privilégie la non-altération des flux.

Filtrage et performance :

  • Filtres de capture (BPF) : limiter les volumes.

  • Éviter le packet loss sur interfaces rapides.

  • Segmentation des fichiers .pcap.

  • Déclencheurs d'enregistrement (Zeek, Suricata).

Stockage & rotation :

  • Rotation automatique avec dumpcap/tcpdump.

  • Nomination horodatée des fichiers.

  • Conservation légale et éthique des captures.

  • Sécurisation (chiffrement, checksums, accès restreint).

Intégrité des preuves réseau :

  • Utiliser des hashs SHA256 pour les .pcap.

  • Chaîne de possession (Chain of Custody).

  • Journalisation des accès aux fichiers.

  • Conformité RGPD et pratiques internes.

Quiz / Discussion / Réflexion :

  1. Comment garantir qu'une capture réseau n'a pas été altérée ?

    En pratique, l'intégrité repose sur la combinaison "hash + horodatage + stockage imuable + traçabilité", pas sur un seul mécanisme.

    Pour garantir qu'une capture réseau n'a pas été altérée, on s'appuie surtout sur des mécanismes d'intégrité et de traçabilité :

    • Hachage (SHA-256, etc.) : calcul d'empreintes du fichier PCAP à la création et à chaque transfert.

    • Chain of custody (chaîne de preuve) : documentation de qui a collecté, manipulé et stocké la capture.

    • Horodatage fiable : synchronisation NTP ou horodatage signé pour prouver le moment de capture.

    • Stockage immmuable : systèmes WORM (Write Once Read Many) ou stockage sécurisé en lecture seule.

    • Signature numérique : signature des fichiers ou des exports de capture.

    • Contrôle d'accès strict : limiter qui peut modifier ou accéder aux fichiers.

    • Journalisation (logs) : traçabilité des accès et opérations sur les captures.

  2. Quelles différences entre filtres de capture et filtres d'affichage ?

    En résumé :

    • filtre de capture = je n'enregistre que ça.

    • filtre d'affichage = je regarde seulement ça.

    Les filtres de capture et les filtres d'affichage (dans des outils comme Wireshark) n'agissent pas au même moment ni sur les mêmes données.

    Les filtres de capture agissent avant ou pendant l'enregistrement des paquets et déterminent directement ce qui sera collecté. Ils permettent de réduire le volume de données dès la source, mais tout paquet non capturé est définitivement perdu et ne pourra plus être analysé par la suite.

    Les filtres d'affichage agissent après la capture et servent uniquement à filtrer les paquets visibles dans l'outil d'analyse. Ils n'altèrent pas les données enregistrées et permettent d'explorer la capture de manière flexible sans risque de perte d'informations.

  3. Comment limiter l'impact d'une capture sur un réseau en production ?

    En résumé, on limite l'impact en capturant moins, plus intelligemment, et en déplaçant l'analyse vers des données agrégées plutôt que du trafic brut.

    Des données agrégées sont des informations résumées issues du trafic réseau, sans conserver les paquets complets.

    Au lieu de capturer chaque paquet (PCAP), on enregistre des statistiques ou métadonnées.

    Pour limiter l'impact d'une capture en production, l'objectif est de réduire la charge sur le réseau, les équipements et la sonde.

    Les principales bonnes pratiques sont :

    • Filtrer à la source : ne capturer que le trafic utile (BPF, filtres précis).

    • Éviter le "full capture" permanent : privilégier des captures ciblées ou temporaires.

    • Utiliser des TAP ou SPAN dédiés : plutôt que de surcharger des ports existants.

    • Limiter le débit de mirroring : éviter la surcharge du port de sortie SPAN.

    • Échantillonnage (sampling) : capturer une partie du trafic seulement.

    • Stockage local rapide : réduire les goulots d'étranglement disque/réseau.

    • Planifier les captures : hors pics de charge si possible.

    • Utiliser des outils légers (eBPF, flow logs) : pour réduire la charge vs PCAP complet.

    • Surveiller les ressources de la sonde : CPU, RAM, buffer drops.

Section 4 : Analyse de trafic & extraction d'indicateurs :

Objectifs :

  • Comprendre comment analyser un fichier PCAP.

  • Identifier des IOC dans le trafic réseau.

  • Reconstruire une chronologie des événements.

  • Détecter les signes d'intrusion ou d'exfiltration.

Outils d'analyse réseau :

  • Wireshark, TShark, NetworkMiner.

  • Zeek pour l'analyse comportementale.

  • Suricata pour l'analyse par signature.

Détection d'anomalies :

  • Ports non standards.

  • Services non autorisés.

  • Volumes anormaux de données.

  • Protocoles non usuels.

Extraction d'IOC :

  • Adresses IP, domaines, User-Agents.

  • Hashes de fichiers (MD5/SHA).

  • Certificats SSL/TLS anormaux.

  • Payloads encodés ou chiffrés.

Chronologie d'attaque :

  • Reconstruction temporelle du trafic.

  • Identification de la machine d'origine.

  • Séquence des actions malicieuses.

  • Corrélation avec logs système.

Quiz / Discussion / Réflexion :

  1. Quelle différence entre IOC statique et comportemental ?

    En résumé :

    • statique = quoi (signature fixe)

    • comportemental = comment (activité suspecte)

    Un IOC (Indicator of Compromise) est un indice technique qui signale une possible compromission d'un système ou d'un réseau.

    Il sert à détecter si une attaque a eu lieu ou est en cours.

    La différence est surtout ce que l'indicateur décrit : une signature fixe ou une activité.

    L'IOC statique identifie quelque chose de figé. Il est facile à détecter, mais facile à contourner (un simple changement casse l'IOC). Par exemple : hash de fichier (SHA256), domaine, nom de malware, signature antivirus.

    L'IOC comportemental décrit une activité ou un pattern suspect. Il est plus robuste, car il détecte le comportement, même si le malware change. Par exemple : connexions répétées à un serveur inconnu, exfiltration progressive de données, processus qui injecte du code, séquence d'actions typique d'un malware.

  2. Peut-on détecter une exfiltration chiffrée ?

    En résumé, on ne voit pas le contenu chiffré, mais on détecte une exfiltration par ses signaux réseau et comportementaux.

    Oui, mais pas en lisant le contenu, puisqu'il est chiffré. On la détecte surtout par indices indirects.

    Les principales méthodes sont :

    • Analyse des volumes : gros transferts sortants inhabituels.

    • Flux réseau (NetFlow/IPFIX) : connexions longues ou répétées vers un même hôte externe.

    • DNS et SNI : domaines rares, nouveaux ou suspects.

    • Comportement : activité anormale d'un poste (par exemple : serveur qui envoie beaucoup de données).

    • Timing : exfiltration régulière (beaconing, envoi périodique).

    • Destination : connexions vers pays ou ASN inhabituels.

    • Fingerprint TLS (JA3/JA4) : clients ou outils réseau suspects.

    • Corrélation SIEM/EDR : activité réseau + processus local.

  3. Quels outils permettent la reconstruction de fichiers ?

    En résumé, les outils de reconstruction reconstituent des fichiers à partir de flux réseau, mais cela fonctionne surtout sur du trafic non chiffré ou déchiffré.

    La reconstruction de fichiers à partir de trafic réseau (ou de captures PCAP) consiste à réassembler des données transportées sur le réseau pour retrouver des fichiers (HTTP, FTP, SMB, etc.).

    Les outils courants sont :

    • Wireshark permet de suivre des flux et exporter certains objets (HTTP, SMB, etc.).

    • tcpflow reconstruit les flux TCP en fichiers lisibles.

    • Zeek extrait automatiquement fichiers, logs et métadonnées réseau.

    • NetworkMiner reconstruit fichiers, images, emails depuis un PCAP.

    • Arkime permet de rechercher et extraire des sessions complètes.

    Les protocoles où la reconstruction est possible sont :

    • HTTP (téléchargements)

    • FTP

    • SMB/CIFS

    • SMTP (emails avec pièces jointes)

    • parfois DNS (exfiltration encodée)

    Les limites importantes sont :

    • TLS 1.3 rend la reconstruction impossible sans déchiffrement

    • fragmentation et multiplexing compliquent la reconstitution

    • pertes de paquets = fichiers incomplets

    • QUIC/HTTP/3 limite fortement l'analyse classique

Section 5 : Protocoles, signatures et anomalies :

Objectifs :

  • Maîtriser les protocoles réseaux standards.

  • Détecter les usages détournés ou suspects.

  • Utiliser les signatures et anomalies pour alerter.

  • Configurer des outils IDS/NSM.

Protocoles à surveiller :

  • HTTP/S - User-Agent, URI, host.

  • DNS - requêtes fréquentes, domaines NXDOMAIN.

  • SMTP - en-têtes, pièces jointes.

  • FTP/SSH - connexions inhabituelles.

Anomalies réseau typiques :

  • Scan de ports, reconnaissance réseau.

  • Taux élevé de paquets SYN ou RST.

  • Flux internes vers IP externes inhabituelles.

Utilisation de signatures IDS :

  • Snort, Suricata : règles et signatures.

  • Exemples de règles : shellcode, exploitation CVE.

  • Alertes : log, drop, reject.

Signature vs Anomalie :

  • Signature : spécifique, faible faux positifs.

  • Anomalie : plus générique, détection d'inconnus.

  • Approche hybride recommandée.

Section 6 : Reconstruction et corrélation :

Objectifs :

  • Reconstituer les sessions TCP et flux applicatifs.

  • Corréler les événements entre plusieurs sources.

  • Identifier des patterns d'attaque.

  • Utiliser les logs réseau pour enrichir une investigation.

Reconstruction de sessions :

  • Suivi de flux TCP : 3-way handshake.

  • Suivi de sessions HTTP, FTP, SMTP.

  • Reconstitution de fichiers ou conversations.

Corrélation avec autres logs :

  • Firewall, EDR, SIEM.

  • Combinaison IP + timestamp.

  • Corrélation temporelle & spatiale.

Timeline d'incident :

  • Fusion des événements pour visualisation.

  • Outils : Timesketch, Elastic, Splunk.

  • Détection des phases MITRE ATT&CK.

Section 7 : Forensique réseau en environnement cloud :

Objectifs :

  • Comprendre les limites d'analyse réseau dans le cloud.

  • Utiliser les outils natifs pour capturer/analyser le trafic.

  • Évaluer les contraintes de confidentialité et sécurité.

Défis du cloud :

  • Pas d'accès direct aux interfaces.

  • Chiffrement TLS par défaut.

  • Flux internes non visibles.

Outils natifs cloud :

  • AWS VPC Traffic Mirroring, GuardDuty.

  • Azure NSG Flow Logs, Network Watcher.

Quiz / Discussion / Réflexion :

  1. Comment assurer la recevabilité d'un rapport forensique ?

    En résumé, un rapport forensique est recevable s'il prouve que les données sont intactes, traçables et analysées de manière rigoureuse et reproductible.

    Pour qu'un rapport forensique soit recevable juridiquement, il doit garantir surtout la fiabilité, la traçabilité et l'intégrité des preuves.

    Les points essentiels sont :

    • Chaîne de conservation (chain of custody) : documenter chaque personne ayant manipulé la preuve.

    • Intégrité des données : hachage (par exemple : SHA-256) des fichiers et vérification régulière.

    • Méthodologie reproductible : expliquer clairement les étapes pour que l'analyse puisse être refaite.

    • Outils fiables et reconnus : utiliser des outils validés (forensic standards).

    • Horodatage précis : synchronisation fiable des systèmes (NTP, logs).

    • Non-altération des preuves : travailler sur des copies (images disque, PCAP), jamais sur l'original.

    • Documentation complète : logs, actions réalisées, paramètres utilisés.

    • Séparation des rôles : limiter les accès et éviter les manipulations non tracées.

Netflow : C'est quoi NetFlow ?

Définition :

Protocole développé par Cisco qui capture des métadonnées sur les communications réseau.

Ne capture pas les paquets eux-mêmes, mais enregistre les caractéristiques du flow :

  • Qui a communiqué avec qui ?

  • Quand ? Combien de temps ?

  • Combien d'octets ont été échangés ?

Netflow : Composants de l'infrastructure NetFlow :

Éléments : Exporter, Collector, Analyzer :

  • Exporter : Génère les flows (exemples : routeur, switch, sonde).

  • Collector : Reçoit les flows et les stocke (exemples : nfcapd, Logstash).

  • Analyzer : Interface de visualisation (exemples : NfSen, ntopng, ELK, Arkime).

  • Netflow v1 : Première version, limitée, pas de timestamp fin, pas de séquence.

  • Netflow v5 : La plus utilisée, stable et supportée, IPv4 uniquement.

  • Netflow v7 : Version améliorée pour Catalyst, peu utilisée.

  • Netflow v8 : Flows agrégés, moins de granularité.

  • Netflow v9 : Template-based, supporte IPv6, MPLS, multicast.

  • IPFIX : Standard IETF basé sur NetFlow v9, extensible.

Structure physique d'une disquette :

  • Pistes (Tracks) :

    • Anneaux concentriques sur le disque.

    • Disque de 3,5 pouces : 80 pistes par face.

  • Secteurs (Sectors) :

    • Divisions de chaque piste.

    • Chaque piste est subdivisée en 18 secteurs sur une disquette de 1,44 Mo.

    • Taille de secteur typique : 512 bytes.

80 pistes * 18 secteurs * 2 têtes * 512 Bytes par secteur = 1474560 Bytes/1024 KB = 1.44 MB.

Concepts de base :

Un volume est une unité de stockage logique qui peut englober une ou plusieurs partitions physiques sur un disque.

Un volume peut être formaté avec un système de fichiers pour stocker des données.

Une partition est une division logique d'un disque physique en sections isolées.

Chaque partition peut être traitée comme un disque distinct et par le système d'exploitation.

Un bloc est la plus petite unité de stockage sur un disque que le système de fichiers peut gérer.

Les fichiers sont stockés en segments de blocs, ce qui facilite l'accès et la gestion des données.
