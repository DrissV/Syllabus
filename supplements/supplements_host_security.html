Instructions SSH pour Windows :

Pour les utilisateurs utilisant Windows (10+) pour l'hôte, voir https://learn.microsoft.com/en-us/windows-server/administration/openssh/openssh-overview pour la configuration du client d'OpenSSH.

Notez que ssh-keyscan retourne un encodage UTF-16 (sur 2 bytes), là où le fichier known_hosts semble devoir être encodé en UTF-8 (1 byte pour ASCII). Pour passer de l'un à l'autre (dans le directory ~\.ssh) :

ssh-keyscan -t ecdsa t.x.y.z > .\known_hosts2

En résumé, cette commande récupère la clé publique SSH du serveur.

gc .\known_hosts2 | sc -Encoding utf8 .\known_hosts

En résumé, cette commande convertit le fichier en UTF-8 pour compatibilité OpenSSH.

On peut après faire un ssh vers t.x.y.z sans recevoir d'avertissement sur l'authenticité du serveur.

Trucs et astuces pour les problèmes de lenteur du démarrage des boîtes Vagrant sur un hôte Windows :

WSL2 :

Apparemment, la coexistance de Vagrant/VirtualBox d'un côté et de Windows WSL2 de l'autre pose problème.

Une possibilité : désactiver complètement WSL2.

Une autre possibilité pour une désactivation temporaire de WSL2 : dans un PowerShell (sans doute admin) :

  • Désactiver (libère VT-x pour VirtualBox) avec la commande suivante et redémarrer ensuite :

    bcdedit /set hypervisorlaunchtype off

  • Réactiver (restaure WSL2, Hyper-V, Docker Desktop) avec la commande suivante et redémarrer ensuite :

    bcdedit /set hypervisorlaunchtype auto

En pratique, on peut également faire deux entrées Windows dans le bootloader avec des configurations différentes :

  • Windows 1 : avec WSL / Hyper-V → pour Docker Desktop.

  • Windows 2 : sans Hyper-V → pour VMware/VirtualBox.

Il ne s'agit pas réellement d'un "dual boot" classique. La solution consiste simplement à créer une seconde entrée de démarrage Windows avec Hyper-V désactivé.

La commande suivante affiche les entrées de boot actuelles :

bcdedit

La commande suivante crée une copie de l'entrée Windows actuelle et va retourner un GUID sous la forme {1f3c1c9e-xxxx-xxxx-xxxx-xxxxxxxxxxxx} :

bcdedit /copy {current} /d "Windows sans Hyper-V (GNS3 / VMware)"

La commande suivante désactive Hyper-V sur cette entrée. Il ne faut pas oblier de remplacer {GUID} par celui reçu :

bcdedit /set {GUID} hypervisorlaunchtype off

On va renommer l'entrée actuelle (WSL / Docker) avec la commande suivante :

bcdedit /set {current} description "Windows avec Hyper-V (WSL / Docker)"

On peut également choisir l'entrée par défaut avec la commande suivante :

bcdedit /default {GUID}

On peut aussi modifier le temps du menu de boot avec la commande suivante (par exemple : menu visible 5 secondes) :

bcdedit /timeout 5

On peut également vouloir que l'entrée sans Hyper-V soit la première ligne du menu de boot avec la commande suivante :

bcdedit /displayorder {current} {GUID}

La configuration est maintenant finie. On va vérifier qu'on a bien deux entrées Windows dans le bootloader avec la commande suivante :

bcdedit

On devrait maintenant voir :

Windows Boot Manager

Windows sans Hyper-V (GNS3 / VMware)
Windows avec Hyper-V (WSL / Docker)

Autorisation :

Concept : Qui peut faire quoi. En pratique, mis en oeuvre via le contrôle d'accès aux ressources.

Les types de contrôle d'accès sont :

  • Autoriser.

  • Interdire.

  • Limiter.

  • Révoquer.

Mise en oeuvre des contrôles d'accès :

Access Control List :

Pour une ressource donnée, qui peut faire quoi.

  • Fichiers.

  • Accès au réseau.

Capabilities - peu fréquent au niveau OS :

L'utilisateur possède un token. Accès aux resources basé uniquement sur la possession du token, pas de celui qui le possède.

Modèles de contrôle d'accès :

Qui détermine l'accès aux ressources.

  • Discretionary Access Control : Le propriétaire de la ressource.

  • Mandatory Access Control : Un groupe séparé détermine l'accès aux ressources.

  • Rule-based, Role-based, Multi-level Access Control : Souvent, pour environnements "sensibles".

Implémentation de contrôle d'accès type UNIX/Linux :

Principaux mécanismes - Evi Nemeth & others :

  • L'accès à une ressource se fait en contrôlant l'utilisateur essayant d'y accéder (et / ou le groupe de l'utilisateur).

  • Les ressources (fichier, processus) ont un propriétaire qui ont un accès large à la ressource.

  • Les propriétaires "possèdent" les ressources qu'ils créent.

  • L'utilisateur "root" peut agir comme le propriétaire de tout objet/ressource.

  • Certaines fonctions administratives sont réservées à l'utilisateur "root".

Extensions :

Limites du modèle type - Evi nemeth & others :

  • Root - point faible crucial : sudo ? - d'une certaine manière, étend la surface d'attaque.

  • suid - division des pouvoirs de Root, mais très vulnérable.

Le fichier /etc/shadow contient les informations sensibles li&es aux mots de passe et à leur expiration.

Chaque ligne correspond à un utilisateur et comporte 9 champs séparés par des : :

login:password:lastchg:min:max:warn:inactive:expire:reserved

Voici l'explication de chaque champ :

  • user.

  • hash :

    • version

      • $6$ → SHA-512 (le plus courant).

      • $5$ → SHA-256.

      • $1$ → MD5.

    • salt = valeur aléatoire ajoutée avant le hash.

    • hash proprement dit.

    Cas particuliers :

    • ! → compte verrouillé.

    • * → pas de connexion possible.

    • vide → pas de mot de passe.

  • diverses informations relatives au mot de passe (dernier changement, etc.).

    • Nombre de jours depuis le 1er janvier 1970 (Epoch) correspondant à la date de la dernière modification.

    • Nombre minimum de jours avant de pouvoir changer le mot de passe.

    • Nombre maximum de jours avant expiration.

    • Nombre de jours d'avertissement avant expiration.

    • Nombre de jours après expiration avant désactivation du compte.

    • Date d'expiration du compte (jours depuis 1970).

    • Champ réservé (généralement vide).

Notions :

Notion & Motivation :

  1. Notion :

    • Mêmes utilisateurs/'credentials' sur toutes les machines du réseau.

    • (Single Sign On - SSO) Ne s'identifier/s'authentifier qu'une seule fois pour une période de temps.

  2. Motivations Administrateur / Responsable Sécurité :

    • Gestion intégrée des utilisateurs et des droits, notamment utilisateurs avec accès privilégié (sudo).

    • Politique mots de passe etc. unifiée...

    • ... mais point de faiblesse unique si compte compromis.

    • (SSO) Centraliser au niveau du réseau l'authentification - pas de compromission des mots de passe si un service est compromis.

  3. Motivation Utilisateurs :

    • Exigence de mots de passe forts plus supportable.

    • (SSO) Facilité d'utilisation.

Composants :

  1. Interface client :

    • sswitch & Pluggable Authentication Module (PAM - Encore lui !) - où trouver l'info & configurer l'authentification.

    • System Security Services Daemon (SSSD) - tâche de fond.

  2. Gestion centralisée de l'identité :

    • Le plus souvent, Lightweight Directory Access Protocol (LDAP).

  3. Gestion centralisée de l'authentification :

    • Directement sur le serveur d'identité LDAP - pas de SSO.

    • Avec Kerberos - nécessaire pour SSO.

Pourquoi Kerberos ? Le problème de l'authentification LDAP seule :

Sans Kerberos, le service voit le mot de passe.

  • L'utilisateur envoie son mot de passe au service.

  • Le service le transmet au serveur LDAP pour vérification.

  • Service compromis = mot de passe capturé et réutilisable partout.

  • L'utilisateur doit retaper son mot de passe pour chaque service sur chaque machine.

Avec Kerberos : le service ne voit jamais le mot de passe.

  • Le mot de passe n'est utilisé qu'une seule fois, entre l'utilisateur et le tiers de confiance.

  • Les services reçoivent des tickets - jamais le mot de passe.

  • Servoce compromis = rien de réutilisable (tickets à durée à durée limitée, liés à un service).

Serveur d'identité (et éventuellement d'authentification) :

LDAP :

  • Un protocole pour accéder à un annuaire via TCP/IP.

  • Annuaire peut être général, mais en pratique bien adapté pour données d'identité et d'authentification.

  • Accès idéalement sécurisé via TLS (certainement si Authentification).

ldapsearch -x -LLL -b dc=hs,dc=be '(uid=john)'
ldapsearch -x -LLL -b dc=hs,dc=be '(cn=john)'

En résumé, ldapsearch est un outil pour interroger un annuaire LDAP. -x utilise l'authentification simple, -LLL produit une sortie épurée, -b dc=hs,dc=be définit la base DN, et '(uid=john)' ou '(cn=john)' est le filtre de recherche.

Authentification réseau - Single Sign On :

Kerberos : authentification via tiers de confiance et "session keys" :

Principes de fonctionnement :

Tiers de confiance :

  • Authentification préalable des utilisateurs (et des services !) auprès d'un tiers de confiance.

Échanges utilisateur - serveur - version simplifiée(!) :

  1. L'utilisateur reçoit du tiers de confiance :

    • un secret ("session key") encrypté avec le mot de passe de l'utilisateur.

    • un "ticket" qui contient le secret, encrypté avec le mot de passe du service.

  2. L'utilisateur décrypte le secret.

  3. L'utilisateur encrypte une requête contenant un "timestamp" encrypté avec le secret et l'envoie au service, en même temps que le "ticket".

  4. Le service décrypte le ticket, avec son propre mot de passe, et utilise le secret pour décrypter la demande de l'utilisateur.

  5. Seul un utilisateur valide peut produire une requête valide - il doit avoir une connaissance du secret commun entre lui et le service.

  6. ... à aucun moment le mot de passe n'est échangé en clair.

  7. ... à aucun moment le service n'a connaissance du vrai mot de passe de l'utilisateur !

  8. Le secret et ticket n'ont qu'une durée de vie limitée.

Single Sign On :

  1. Le secret initial peut servir comme "credential" entre l'utilisateur et un service spécial et centralisé qui crée des "sous-secrets et sous-tickets".

  2. Ces "sous-secrets et sous-tickets" sont utilisés par l'utilisateur pour accéder à d'autres services, sans redonner son mot de passe - single sign on !

  3. Le secret initial a de toute façon une durée de vie limitée.

Très bonne explication : https://wpollock.com/AUnixSec/MoronsGuideToKerberos.htm

Single Sign On - vue utilisateur :
L'expériece au quotidien :

  1. Je tape mon mot de passe une seule fois au login.

  2. J'obtiens un TGT (Tocket-Granting Ticket) - mon "passe" pour la session.

  3. J'accède aux services réseau (SSH, web, fichiers...) sans retaper mon mot de passe.

  4. Après expiration du TGT (typiquement 10h). Il faut se réauthentifier.

Single Sign On - vue sécurité :
Les garanties :

  1. Le mot de passe ne circule jamais sur le réseau (contrairement à l'authentification LDAP directe).

  2. Les tickets sont à durée limitée - un ticket volé a une fenêtre d'exploitation réduite.

  3. Authentification mutuelle - le service prouve son identité au client (protection MITM).

  4. Un service compromis ne capture rien de réutilisable.

Kerberos - Phase 1 : Authentification initiale (AS Exchange) :

john@sso-client → AS sur sso-server :

  1. john envoie à l'AS : Je suis john.

  2. L'AS répond avec :

    • une clé de session TGS, chiffrée avec le mot de passe de john.

    • un TGT (Ticket-Granting Ticket), chiffré avec la clé du TGS (john ne peut pas le le lire - c'est un "message scellé" pour le TGS).

  3. john déchiffre la clé de session avec son mot de passe.

  4. john stocke le TGT + la clé de session dans un cache local (un "portefeuille de tickets").

Point clé : Le mot de passe a servi une seule fois. Tout le reste repose sur le TGT et la clé de session.

Kerberos - Phase 2 : Accès au service (TGS + AP Exchange) :

john@sso-client → TGS sur sso-server → sshd sur sso-service :

  1. john envoie au TGS : TGT + authenticator * + je veux host/sso-service.hs.be.

  2. Le TGS déchiffre le TGT avec sa propre clé, vérifie l'authenticator.

  3. Le TGS répond avec :

    • une clé de session service, chiffrée avec la clé de session TGS.

    • un service ticket, chiffré avec la clé de host/sso-service.hs.be.

  4. john envoie au sshd : service ticket + authenticator.

  5. Le sshd déchiffre le service ticket avec son keytab, vérifie l'authenticator.

  6. Le sshd répond → authentification mutuelle.

* authenticator = timestamp chiffré avec la clé de session - prouver que john la possède maintenant (protection contre le rejeu).

Kerberos - Parallélisme structurel des deux phases :
Phase 1 (AS) Phase 2 (TGS)
John prouve son identité avec son mot de passe TGT + authenticator *
Le serveur chiffre le ticket avec clé du TGS (krbtgt) clé de host/sso-service
John reçoit TGT + clé de session service ticket + clé de session
Ce ticket sera présenté à le TGS sshd sur sso-service

* authenticator = timestamp chiffré avec la clé de session, preuve de possession "en temps réel".

Le TGS est "juste un service".

La seule différence : la première fois, john tape son mot de passe. Les fois suivantes, le TGT le remplace. C'est ça, le Single Sign-On.

Authentification mutuelle - le service prouve son identité :

Le problème est :

  • Un attaquant pourrait se faire passer pour sso-service.hs.be (attaque MITM).

  • Le client enverrait son service ticket à l'imposteur.

La solution est le keytab :

  • Le service ticket est chiffré avec la clé de host/sso-service.hs.be.

  • Seul le vrai sso-service possède cette clé (dans son keytab : /etc/krb5.keytab).

  • Le service déchiffre le ticket, extrait la clé de session, construit une réponse.

  • Le client vérifie cette réponse → le service a prouvé son identité.

  • Un imposteur sans keytab ne peut pas produire cette réponse.

Le keytab - "mot de passe" de la machine :
Création (sur sso-server, lors du provisioning) :

Voici les différentes commandes exécutées dans l'image précédente :

kadmin.local -q "addprinc -randkey host/sso-service.hs.be"

En résumé, cette commande crée un principal Kerberos pour le service avec une clé secrète générée aléatoirement dans la base du KDC.

kadmin.local -q "ktadd -k /etc/krb5.keytab host/sso-service.hs.be"

En résumé, cette commande exporte la clé du principal dans le fichier keytab afin que le service puisse déchiffrer les tickets Kerberos.

Explication :

  • addprinc -randkey : crée un principal avec un mot de passe aléatoire.

  • ktadd : exporte cette clé dans un fichier keytab.

  • Secret partagé entre la base du KDC (côté sso-server) et /etc/krb5.keytab (côté sso-service).

  • Analogie: le keytab est au service ce que le mot de passe est à l'utilisateur.

Sans keytab, le sshd ne peut pas déchiffrer le service ticket → authentification GSSAPI impossible.

  • Limite : ne prouve pas qui a produit le message.

MAC - Message Authentication Code :

  • Quoi ? Hash calculé avec un secret partagé entre les deux parties.

  • Pourquoi ? Prouve l'intégrité et que l'émetteur connaît le secret.

  • ≠ hash : sans le secret, impossible de forger le MAC.

  • ≠ signature : pas de clé publique/privée - symétrique, donc rapide.

Signature d'un message avec clé asymétrique :

  • Quoi ? Hash du message chiffré avec la clé privée de l'émetteur.

  • Pourquoi ? Prouve l'origine et l'intégrité du message.

  • Vérification : n'importe qui peut vérifier avec la clé publique.

Problème : Le chiffrement symétrique est rapide, mais comment partager le secret initial entre deux parties qui ne se connaissent pas ?

Solution : combiner les deux → TLS, SSH, VPN :

  1. Crypto asymétrique pour établir un secret partagé (Diffie-Hellman).

  2. Ce secret en crypto symétrique (AES) pour chiffrer - rapide.

  3. MAC (HMAC) pour vérifier l'intégrité.

Étape Rôle Crypto utilisée
Établissement du secret Échange de clé Asymétrique (DH)
Chiffrement des données Confidentialité Symétrique (AES)
Vérification d'intégrité Intégrité MAC (HMAC-SHA256)
Identité du serveur Authenticité Certificat (RSA/ECDSA)

Et concrètement ? Chaque protocole (TLS, SSH, VPN) implémente ce schéma à sa manière. Dans la suite du cours, on va montrer TLS.

SSH : documentation :

  1. Décrire la procédure hors-bande utilisée pour échanger les clés publiques et vérifier la clé hôte :

    La clé publique du client est copiée via la console des VMs, puis collée dans authorize_keys.

    • Pourquoi la console est-elle un canal de confiance ?

      Parce qu'elle n'utilise pas le réseau, donc un attaquant ne peut pas intercepter ou modifier la clé.

    • Que se passerait-il si la clé publique d'un utilisateur était interceptée ou modifier lors de son transfert vers authorized_keys ?

      Un attaquant pourrait remplacer la clé publique par la sienne et obtenir un accès SSH non autorisé.

  2. Expliquer pourquoi chaque utilisateur possède son propre known_hosts : quelles conséquences si un utilisateur accepte une mauvaise clé hôte (celle d'un attaquant) ?

    Chaque utilisateur doit vérifier lui-même la clé du serveur. Si un utilisateur accepte une mauvaise clé, il peut être victime d'une attaque Man-in-the-Middle.

  3. Une fois les clés en place, que vérifie le serveur lors d'une connexion SSH ? Que vérifie le client ? En quoi est-ce une authentification mutuelle ?

    Le serveur vérifie que la clé publique du client correspond à une clé dans authorized_keys.

    Le client vérifie que la clé du serveur correspond à celle stockée dans known_hosts.

    C'est une authentification mutuelle parce que le serveur prouve son identité au client et que le client prouve son identité au serveur.

  4. Identifier le point commun fondamental entre l'établissement de confiance en SSH (clé hôte, authorized_keys) et en HTTPS (certificat AC) : dans les deux cas, comment l'authenticité est-elle garantie avant la première communication sécurisée ?

    Dans les deux cas, la première vérification de confiance se fait hors-bande. En SSH par la vérification de la clé du serveur et en HTTPS par la confiance dans l'autorité de certification.

VPN :

Vision historique de l'accès à distance :

  • Travail essentiellement sur site.

  • Infrastructure relativement simple.

  • Peu / pas de "cloud".

  • Accès de l'extérieur plutôt exceptionnel.

  • Approche périmétrique de la sécurité extérieure = danger (et souvent implicite : intérieur = sûr).

Cas d'utilisation :

  • Contexte :

    • Liaison de serveurs d'entreprise distants.

    • Connexion au réseau d'entreprise dans le cadre du télétravail.

    • (À priori, identité des hôtes "facile" à établir.)

  • Motivation :

    • Confidentialité.

    • Intégrité.

    • Prévention MITM : s'assurer de la bonne identité des hôtes.

SSH : avant tout administration système d'une machine (shell, SCP/SFTP); un tunnel par port (-L) ou proxy SOCKS(-D) peut exposer d'autres services, mais chaque service nécessite une configuration explicite.

VPN : transparence au niveau réseau - toutes les applications atteignent directement les hôtes distants, sans modification.

WireGuard - Étapes cryptographiques :

Étape Rôle Mécanisme
Auth. mutuelle Identité des deux pairs Clés publiques pré-configurées (Curve25519)
Échange de clé éphémère Secret partagé DH éphémère (Curve25519)
Chiffrement + intégrité Confidentialité + intégrité ChaCha20-Poly1305 (AEAD)

Authentification mutuelle par conception : Chaque pair configure explicitement la clé publique de l'autre (PublicKey dans wg0.conf). Un pair inconnu est silencieusement ignoré : pas de PKI, pas de CA.

Forward secrecy : clés éphémères jetées après chaque session - une fuite future de la clé statique ne compromet pas le trafic passé.

AEAD (ChaCha20-Poly1305) : chiffrement + intégrité en un seul primitif (contrairement à AES + HMAC séparés dans TLS/SSH).

Concept : chiffrement au niveau bloc :

  • dm-crypt au niveau du noyau : toute lecture/écriture au niveau bloc est médiatisée par une étape crypto (transparent pour le système de fichiers).

  • LUKS/cryptsetup au niveau utilisateur : interface de gestion (formatage, ouverture, fermeture des volumes).

  • Attention : données accessibles en mode suspend (clé en RAM, volume monté).

Architecture LUKS - clé maître et key slots :

  • Le volume est chiffré avec une clé maître (master key) aléatoire, jamais exposée directement à l'utilisateur.

  • La passphrase ne chiffre pas le volume - elle chiffre la master key, stockée dans un key slot (LUKS2 : jusqu'à 32 slots).

Conséquences pratiques :

  • Changer la passphrase → rechiffre seulement la master key (millisecondes, volume intact).

  • Passphrase compromise → il faut rechiffrer tout le volume (cryptsetup-reencrypt).

  • Plusieurs passphrases en parallèle : utilisateur + clé de récupération admin dans deux slots distincts.

Structure du header LUKS :

Key slot 0 : passphrase utilisateur.

Key slot 1 : clé de récupération.

... (jusqu'à 32).

← chaque slot contient la master key chiffrée.

Volume : chiffré AES-XTS avec la master key.

Comparaison : chiffrement fichier/FS vs disque entier :

  • Fichier/FS :

    • Granularité : protection ciblée de certains fichiers.

    • Facilité de sauvegarde (backup).

    • Surtout pertinent sur une machine patagée.

  • Disque entier :

    • Protection contre le vol physique.

    • Toutes les partitions déverrouillées en une seule opération.

    • Sauvegardes plus complexes (image chiffrée).

    • "Disque entier" est relatif : possible de ne chiffrer qu'une partition.

    • (en pratique, la partition /boot reste non chiffrée).

  • Les deux approches peuvent se combiner.

Chiffrement du swap :

  • Données sensibles normalement pas copiées dans le swap...

  • ... mais le noyau ne peut pas toujours garantir l'absence de fuite.

  • Solution 1 : partition swap séparée, chiffrée avec dm-crypt et une clé éphémère (générée à chaque démarrage) → pas d'hibernation possible.

  • Solution 2 : partition swap dans le volume LUKS → permet l'hibernation (image chiffrée sur disque).

Limites et compléments :

  • Cold boot attack : après arrêt brutal, des rémanences de la RAM permettent d'extraire les clés (quelques secondes à minutes selon la température).

  • Suspend vs Hibernate :

    • suspend : clés en RAM, volume monté - non protégé.

    • hibernate : état sauvegardé sur disque - protéger si swap chiffré.

  • TPM : puce matérielle stockant la clé LUKS liée au matériel - déverrouillage automatique au boot sans passphrase (BitLocker, LUKS + clevis).

    → compromis commodité / protection contre vol du seul disque.

  • Rotation des clés : passphrase compromise → changer le mot de passe ne suffit pas - rechiffrer tout le volume (cryptsetup-reencrypt).

Découverte du monde du pentesting physique et sécurisation des infrastructures :

Introduction :

La sécurité informatique ne se limite pas uniquement aux systèmes logiciels ou aux réseaux. Elle inclut également la sécurité physique des infrastructures, souvent négligée, mais tout aussi critique. Le pentesting physique consiste à simuler des attaques réelles afin d'identifier les failles dans la protection des bâtiments, des équipements et des utilisateurs. À travers l'analyse de la vidéo "I'll Let Myself In : Tactics of Physical Pen Testers", j'ai pu comprendre les techniques utilisées par les pentesters, leurs priorités, les outils employés ainsi que les contre-mesures permettant d'améliorer la sécurité.

Techniques utilisées par les pentesters physiques :

Les pentesters physiques utilisent un ensemble de techniques variées pour pénétrer dans des environnements sécurisés. L'une des méthodes les plus efficaces est le social engineering, qui consiste à exploiter le comportement humain plutôt que des failles techniques. Par exemple, un pentester peut se faire passer pour un technicien de maintenance, un livreur ou un employé afin d'obtenir un accès sans autorisation. Cette technique repose sur la confiance, la pression sociale ou le manque de vigilance.

Une autre technique courante est le tailgating, qui consiste à suivre une personne autorisée pour entrer dans une zone sécurisée sans utiliser de badge. Cette méthode fonctionne particulièrement bien dans les environnements où les employés ne vérifient pas systématiquement les accès.

Les pentesters peuvent également recourir à des techniques plus techniques, comme le crochetage de serrures (lockpicking) ou l'utilisation d'outils permettant de contourner les systèmes de sécurité physique. Le clonage de badges RFID est également une méthode fréquente, permettant de reproduire un badge d'accès légitime.

Une fois à l'intérieur, les pentesters cherchent à exploiter les accès disponibles. Ils peuvent brancher des dispositifs sur le réseau, déposer des clés USB piégées ou accéder directement à des postes de travail non verrouillés. Ces actions permettent de compromettre les systèmes informatiques sans recourir à des attaques réseau classiques.

Priorités des pentesters physiques :

Les pentesters physiques adoptent une approche stratégique basée sur l'efficacité et la discrétion. Leur objectif est de réussir l'intrusion avec le minimum d'effort et de risque.

La priorité est donnée aux failles humaines. En effet, les utilisateurs représentent souvent le point le plus faible d'un système de sécurité. Il est généralement plus simple de manipuler une personne que de contourner un dispositif technique complexe. Par exemple, convaincre un employé de laisser passer quelqu'un ou de fournir des informations confidentielles est souvent plus rapide que de forcer une porte sécurisée.

Ensuite, les pentesters exploitent les failles organisationnelles et physiques, telles que des portes mal verrouillées, des badges laissés sans surveillance ou des équipements accessibles librement. Ces erreurs sont fréquentes dans les environnements professionnels.

Les techniques plus avancées, comme le crochetage ou l'utilisation d'outils spécialisés, ne sont utilisées qu'en dernier recours. Cette approche reflète la réalité des attaques, où les attaquants privilégient toujours les méthodes les plus simples et les moins détectables.

Outils et techniques utilisés :

Les pentesters utilisent divers outils spécialisés pour faciliter leurs actions. Certains de ces outils sont commercialisés par des entreprises comme Hak5, qui proposent du matériel dédié aux tests de sécurité.

Parmi les outils les plus utilisés, on retrouve :

  • USB Rubber Ducky : un périphérique qui se comporte comme un clavier et exécute automatiquement des commandes malveillantes dès qu'il est connecté.

  • WiFi Pineapple : un appareil permettant d'intercepter les connexions Wi-Fi et de réaliser des attaques de type "man-in-the-middle".

  • LAN Turtle : un dispositif discret qui se branche sur un réseau et permet un accès à distance.

  • Keylogger USB : un outil capable d'enregistrer toutes les frappes clavier.

  • Proxmark3 : utilisé pour analyser et cloner des badges RFID.

  • Bash Bunny : un outil avancé permettant d'automatiser des attaques via USB.

Ces outils permettent de réaliser différentes attaques, comme l'injection de commandes, la surveillance réseau ou la récupération d'informations sensibles. Ils sont souvent utilisés en complément des techniques de social engineering pour maximiser les chances de réussite.

Contre-mesures pour améliorer la sécurité physique :

Pour se protéger contre les attaques physiques, il est nécessaire de mettre en place une stratégie de sécurité globale.

Tout d'abord, la sécurisation des accès physiques est essentielle. Cela inclut l'utilisation de badges sécurisés, de systèmes d'authentification à plusieurs facteurs et de dispositifs empêchant le tailgating, comme les sas de sécurité. Les portes doivent être correctement verrouillées et contrôlées.

Ensuite, la surveillance joue un rôle important. L'installation de caméras de sécurité, d'alarmes et de systèmes de contrôle d'accès permet de détecter et de dissuader les intrusions.

La sensibilisation des employés est également un élément clé. Les utilisateurs doivent être formés aux risques liés au social engineering et apprendre à vérifier l'identité des personnes avant de leur accorder un accès. Des procédures strictes doivent être mises en place pour encadrer les accès aux zones sensibles.

Sur le plan informatique, il est recommandé de désactiver ou de contrôler les ports USB, de surveiller le trafic réseau et de limiter les droits d'accès des utilisateurs. Les postes de travail doivent être verrouillés en cas d'absence.

Enfin, les infrastructures critiques, comme les salles serveurs, doivent bénéficier d'une protection renforcée. L'accès doit être limité, surveillé et journalisé afin de garantir une traçabilité complète des actions.

Conclusion :

Le pentesting physique met en évidence l'importance d'une approche globale de la sécurité. Les attaques ne reposent pas uniquement sur des vulnérabilités techniques, mais aussi sur des failles humaines et organisationnelles. Une sécurité efficace doit donc combiner des mesures physiques, techniques et humaines. En renforçant la vigilance des utilisateurs et en améliorant les dispositifs de protection, il est possible de réduire significativement les risques d'intrusion et de compromission des systèmes.

# Reject all other inbound.
-A INPUT -j REJECT

# Log any traffic that was sent to you
# for forwarding (optional but useful).
-A FORWARD -m limit --limit 5/min -j LOG --log-prefix "iptables_FORWARD_denied: " --log-level 7

# Reject all traffic forwarding.
-A FORWARD -j REJECT

COMMIT

La ligne *filter dit qu'on travaille dans la table filter (la plus classique) qui, elle, sert à autoriser/bloquer le trafic.

La ligne -A INPUT -i lo -j ACCEPT indique qu'on autorise tout le trafic venant de localhost (lo).

La ligne -A INPUT ! -i lo -s 127.0.0.0/8 -j REJECT permet de bloquer toute tentative d'usurpation. C'est-à-dire, si une IP 127.X.X.X n'arrive pas via lo = spoofing → rejeté.

La ligne -A INPUT -p icmp -m state --state NEW --icmp-type 8 -j ACCEPT autorise les pings entrants (icmp-type 8 = echo request).

La ligne -A INPUT -p tcp --dport 22 -m state --state NEW -j ACCEPT dit qu'on autorise les connexions SSH (port 22).

Les lignes -A INPUT -p tcp --dport 80 -m state --state NEW -j ACCEPT et -A INPUT -p tcp --dport 443 -m state --state NEW -j ACCEPT autorisent respectivement HTTP (80) et HTTPS (443). Donc, ton serveur web est accessible.

La ligne -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT permet d'autoriser les réponses à des connexions déjà ouvertes et les connexions liées. Super important sinon rien ne fonctionne correctement.

La ligne -A INPUT -m limit --limit 5/min -j LOG --log-prefix "iptables_INPUT_denied: " --log-level 7 indique qu'on log les paquets refusés (max 5/min). C'est utile pour le débogage.

La ligne -A INPUT -j REJECT dit que tout ce qui n'est pas autorisé est rejeté. Politique : deny by default.

La ligne -A FORWARD -m limit --limit 5/min -j LOG --log-prefix "iptables_FORWARD_denied: " --log-level 7 permet de logger les paquets en transit refusés.

La ligne -A FORWARD -j REJECT bloque tout routage. Donc, la machine ne sert pas de routeur.

La ligne COMMIT dit qu'on applique toutes les règles.

Machine virtuelle maintenance+sec - architecture :

Chaque couche tourne sous un utilisateur OS distinct. Les communications inter-couches passent par des sockets Unix - aucun port TCP interne. Objectif : si une couche est compromise, les autres restent protégées.

Sockets Unix - principe :

TCP/IP loopback :

nginx → 127.0.0.1:9000 → php-fpm

  • Port ouvert (visible dans ss).

  • Pile réseau complète traversée.

  • Toute appli peut tenter de s'y connecter.

Sandboxing systemd - principe :

systemd peut restreindre ce qu'un service peut faire, indépendamment des droits Unix. Une faille dans nginx n'ouvre pas l'ensemble du système.

Directive Effet
NoNewPrivileges=yes Interdit setuid, escalade de privilèges
PrivateTmp=yes /tmp isolé (invisible des autres services)
ProtectSystem=yes /usr, /boot en lecture seule
ProtectHome=yes /home, /root invisibles
RestrictAddressFamilies Familles réseau autorisées seulement
DynamicUser=yes UID aléatoire à chaque démarrage (avancé)

Fichier : /etc/systemd/system/<service>.d/sandbox.conf (surcharge de l'unité livrée par le paquet, sans le modifier).

Vérification - systemd-analyze security :

Sécurisation des hôtes (Linux) : AppArmor : contrôle d'accès obligatoire :

DAC (Unix classique) :

  • Droits définis par le propriétaire du fichier.

  • Un process root peut tout faire.
